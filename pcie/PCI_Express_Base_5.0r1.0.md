## 第三章 数据链路层规范

数据链路层作为事务层和物理层之间的中间阶段。其主要职责是提供一种可靠的机制，用于在链路上的两个组件之间交换事务层数据包 (TLP)。

### 3.1 数据链路层概述

数据链路层负责将事务层提供的 TLP 可靠地传输到 PCI Express 链路上的另一组件的事务层。数据链路层提供的服务包括：

**数据交换：**
- 从发送事务层接受 TLP 并将其传送到发送物理层
- 从物理层接受通过链路接收的 TLP 并将其传送到接收事务层

**错误检测和重试：**
- TLP 序列号和 LCRC 生成
- 存储已发送的 TLP 用于数据链路层重试
- TLP 和数据链路层数据包 (DLLP) 的数据完整性检查
- 确认和否定确认 DLLP
- 用于错误报告和记录机制的错误指示
- 链路确认超时重放机制

**初始化和电源管理：**
- 跟踪链路状态并向事务层传递活动/复位/断开状态

**DLLP：**
- 用于链路管理功能，包括 TLP 确认、电源管理和流控信息交换
- 在链路上两个直接连接组件的数据链路层之间传输

DLLP 在一条链路上的两个组件之间点对点发送。TLP 从一个组件路由到另一个组件，可能经过一个或多个中间组件。

DLLP 和 TLP 的数据完整性检查使用随每个数据包发送的 CRC 完成。DLLP 使用 16 位 CRC，而 TLP（可能比 DLLP 长得多）使用 32 位 LCRC。TLP 还包括序列号，用于检测一个或多个完整 TLP 丢失的情况。

未通过 CRC 检查的已接收 DLLP 将被丢弃。使用 DLLP 的机制可能因信息丢失而遭受性能损失，但具有自我修复能力，后续的 DLLP 将取代任何丢失的信息。

未通过数据完整性检查（LCRC 和序列号）或在从一个组件传输到另一个组件过程中丢失的 TLP 将由发送方重新发送。发送方存储所有已发送 TLP 的副本，在需要时重新发送这些副本，并且仅在收到来自另一组件的无错误接收的肯定确认后才清除这些副本。如果在指定时间内未收到肯定确认，发送方将自动开始重新传输。接收方可以使用否定确认请求立即重新传输。

数据链路层对于事务层而言表现为一个具有可变延迟的信息通道。在任何给定的单个链路上，所有输入到发送数据链路层（1 和 3）的 TLP 将在稍后的时间以相同的顺序出现在接收数据链路层（2 和 4）的输出端，如图 3-1 所示。延迟取决于多种因素，包括流水线延迟、链路的宽度和操作频率、电信号在链路上的传输以及数据链路层重试引起的延迟。由于这些延迟，发送数据链路层（1 和 3）可以对发送事务层施加反压，接收数据链路层（2 和 4）向接收事务层传递有效信息的存在或缺失。

### 3.2 数据链路控制和管理状态机

数据链路层跟踪链路的状态。它与事务层和物理层通信链路状态，并通过物理层执行链路管理。数据链路层包含数据链路控制和管理状态机 (DLCMSM) 来执行这些任务。该状态机的状态如下所述，并在图 3-2 中显示。

**状态：**
- **DL_Inactive** - 物理层报告链路不可操作或端口未连接任何设备
- **DL_Feature**（可选）- 物理层报告链路可操作，执行数据链路特性交换
- **DL_Init** - 物理层报告链路可操作，为默认虚拟通道初始化流控
- **DL_Active** - 正常操作模式

**状态输出：**
- **DL_Down** - 数据链路层未与链路另一侧的组件通信
- **DL_Up** - 数据链路层正在与链路另一侧的组件通信

#### 3.2.1 数据链路控制和管理状态机规则

每个状态的规则：

**• DL_Inactive**
- PCI Express 热插拔、热复位或冷复位后的初始状态（见第 6.6 节）。注意：DL 状态不受 FLR 影响（见第 6.6 节）。
- 进入 DL_Inactive 时：
  - 将所有数据链路层状态信息重置为默认值
  - 如果端口支持可选的数据链路特性交换，必须清除 Remote Data Link Feature Supported 和 Remote Data Link Feature Supported Valid 字段
  - 丢弃数据链路层重试缓冲区的内容（见第 3.6 节）
- 在 DL_Inactive 状态期间：
  - 向事务层以及数据链路层的其余部分报告 DL_Down 状态。注意：这将导致事务层丢弃所有未完成的事务并终止内部任何发送 TLP 的尝试。对于下游端口，这等同于"热移除"。对于上游端口，链路断开等同于热复位（见第 2.9 节）。
  - 丢弃来自事务层和物理层的 TLP 信息
  - 不生成或接受 DLLP
- 如果满足以下条件，退出到 DL_Feature：
  - 端口支持可选的数据链路特性交换，Data Link Feature Exchange Enable 位置位，事务层指示链路未被软件禁用，且物理层报告 Physical LinkUp = 1b
- 如果满足以下条件，退出到 DL_Init：
  - 端口不支持可选的数据链路特性交换，事务层指示链路未被软件禁用且物理层报告 Physical LinkUp = 1b；或
  - 端口支持可选的数据链路特性交换，Data Link Feature Exchange Enable 位清除，事务层指示链路未被软件禁用，且物理层报告 Physical LinkUp = 1b

**• DL_Feature**
- 在 DL_Feature 状态期间：
  - 执行第 3.3 节所述的数据链路特性交换协议
  - 报告 DL_Down 状态
  - 处于 DL_Down 状态的端口的数据链路层允许丢弃任何接收到的 TLP，前提是它不通过发送一个或多个 Ack DLLP 来确认这些 TLP
- 如果满足以下条件，退出到 DL_Init：
  - 数据链路特性交换成功完成，且物理层继续报告 Physical LinkUp = 1b；或
  - 数据链路特性交换确定远端数据链路层不支持可选的数据链路特性交换协议，且物理层继续报告 Physical LinkUp = 1b
- 如果满足以下条件，终止数据链路特性交换协议并退出到 DL_Inactive：
  - 物理层报告 Physical LinkUp = 0b

**• DL_Init**
- 在 DL_Init 状态期间：
  - 按照第 3.4 节所述的流控初始化协议，为默认虚拟通道 VC0 初始化流控
  - 在 FC_INIT1 状态时报告 DL_Down 状态；在 FC_INIT2 状态时报告 DL_Up 状态
  - 处于 DL_Down 状态的端口的数据链路层允许丢弃任何接收到的 TLP，前提是它不通过发送一个或多个 Ack DLLP 来确认这些 TLP
- 如果满足以下条件，退出到 DL_Active：
  - 流控初始化成功完成，且物理层继续报告 Physical LinkUp = 1b
- 如果满足以下条件，终止 VC0 流控初始化尝试并退出到 DL_Inactive：
  - 物理层报告 Physical LinkUp = 0b

**• DL_Active**
- DL_Active 被称为正常操作状态
- 在 DL_Active 状态期间：
  - 按照本章规定接受和传输与事务层和物理层的 TLP 信息
  - 按照本章规定生成和接受 DLLP
  - 向事务层和数据链路层报告 DL_Up 状态
- 如果满足以下条件，退出到 DL_Inactive：
  - 物理层报告 Physical LinkUp = 0b
  - 支持意外断开错误报告的下游端口（见第 7.5.3.6 节）必须将此从 DL_Active 到 DL_Inactive 的转换视为意外断开错误，但在以下错误检测被阻止的情况下除外：
    - 如果软件置位了 Bridge Control 寄存器中的 Secondary Bus Reset 位，则后续到 DL_Inactive 的转换不得被视为错误。
    - 如果软件置位了 Link Disable 位，则后续到 DL_Inactive 的转换不得被视为错误。
    - 如果交换机下游端口由于该端口上方的事件而转换到 DL_Inactive，则该到 DL_Inactive 的转换不得被视为错误。示例事件包括交换机上游端口传播热复位、交换机上游链路转换到 DL_Down，以及交换机上游端口中的 Secondary Bus Reset 位置位。
    - 如果 PME_Turn_Off 消息已通过此端口发送，则后续到 DL_Inactive 的转换不得被视为错误。
    - 注意：这种情况下的 DL_Inactive 转换在断电、复位或向物理层发送恢复链路的请求之前不会发生。
    - 还要注意，在 PME_Turn_Off/PME_TO_Ack 握手未能成功完成的情况下，可能会检测到意外断开错误。
  - 如果端口与支持热插拔的插槽相关联（Slot Capabilities 寄存器中的 Hot-Plug Capable 位置位），且 Slot Capabilities 寄存器中的 Hot-Plug Surprise 位置位，则任何到 DL_Inactive 的转换不得被视为错误。
  - 如果端口与支持热插拔的插槽相关联（Slot Capabilities 寄存器中的 Hot-Plug Capable 位置位），且 Slot Control 寄存器中的 Power Controller Control 位置位（断电），则任何到 DL_Inactive 的转换不得被视为错误。

由上述一个或多个情况启动的错误阻止必须在端口退出 DL_Active 并随后返回 DL_Active 时保持有效，且在返回 DL_Active 时没有任何阻止情况生效。

注意，从 DL_Active 的转换只是按错误检测阻止条件预期的预期转换。

如果实现，这是与检测端口相关的报告错误（见第 6.2 节）。

**实现说明：物理层节流**

注意，在某些情况下，物理层可能暂时无法接受来自数据链路层的 TLP 和 DLLP。数据链路层必须理解这一点，提供机制让物理层传递此条件，并使 TLP 和 DLLP 在此条件下暂时被阻止。

### 3.3 数据链路特性交换

数据链路特性交换协议是可选的。实现此协议的端口包含数据链路特性扩展功能（见第 7.7.4 节）。此功能包含四个字段：

- **Local Data Link Feature Supported 字段**指示本地端口支持的数据链路特性
- **Remote Data Link Feature Supported 字段**指示远端端口支持的数据链路特性
- **Remote Data Link Feature Supported Valid 位**指示 Remote Data Link Feature Supported 字段包含有效数据
- **Data Link Feature Exchange Enable 字段**允许系统禁用数据链路特性交换。这可用于解决不能正确忽略 DLLP 的遗留硬件问题。

数据链路特性交换协议将端口的本地特性支持信息传输到远端端口，并捕获该远端端口的特性支持信息。

此协议的规则如下：

- 进入 DL_Feature 时：
  - 必须清除 Remote Data Link Feature Supported 和 Remote Data Link Feature Supported Valid 字段
- 在 DL_Feature 状态期间：
  - 事务层必须阻止 TLP 的传输
  - 发送数据链路特性 DLLP
    - 发送的 Feature Supported 字段必须等于 Local Data Link Feature Supported 字段
    - 发送的 Feature Ack 位必须等于 Remote Data Link Feature Supported Valid 位
  - 数据链路特性 DLLP 必须至少每 34 μs 发送一次。在 Recovery 或 Configuration LTSSM 状态中花费的时间不计入此限制
  - 处理接收到的数据链路特性 DLLP：
    - 如果 Remote Data Link Feature Supported Valid 位清除，则将接收到的数据链路特性 DLLP 中的 Feature Supported 字段记录到 Remote Data Link Feature Supported 字段中，并置位 Remote Data Link Feature Supported Valid 位
- 如果满足以下条件，退出 DL_Feature：
  - 收到 InitFC1 DLLP
  - 收到 MR-IOV MRInit DLLP（编码 0000 0001b）；或
  - 在 DL_Feature 状态期间，至少收到一个 Feature Ack 位置位的数据链路特性 DLLP

每个数据链路特性在 Feature Supported 字段中有一个关联位。当该位在 Local Data Link Feature Supported 和 Remote Data Link Feature Supported 字段中都置位时，数据链路特性被激活。

数据链路特性及其对应的位位置如表 3-1 所示。

|位位置|描述|

### 3.4 流控初始化协议

在上电或互连复位后开始正常操作之前，必须为默认虚拟通道 VC0 初始化流控（见第 6.6 节）。此外，当启用额外的虚拟通道 (VC) 时，必须在新启用的每个 VC 能够使用之前完成流控初始化过程（见第 2.6.1 节）。本节描述用于所有 VC 的初始化过程。注意，由于 VC0 在所有其他 VC 之前启用，因此在 VC0 初始化之前不会有任何类型的 TLP 流量。但是，当初始化额外的 VC 时，通常会有 TLP 流量在其他已启用的 VC 上流动。此类流量对额外 VC 的初始化过程没有直接影响。

VC 初始化过程有两个状态。这些状态是：
- FC_INIT1
- FC_INIT2

此过程的规则在以下章节中给出。

#### 3.4.1 流控初始化状态机规则

- 如果在 VC 1-7 的初始化期间任何时候禁用了 VC，则该 VC 的流控初始化过程终止
- FC_INIT1 状态的规则：
  - 当需要初始化 VC (VCx) 时进入：
    - 进入 DL_Init 状态时（VCx = VC0）
    - 软件启用 VC (VCx = VC1-7) 时（见第 7.9.1 节和第 7.9.2 节）
  - 在 FC_INIT1 状态期间：
    - 事务层必须阻止使用 VCx 的 TLP 传输
    - 按以下相对顺序发送 VCx 的以下三个 InitFC1 DLLP：
      - InitFC1-P（第一个）
      - InitFC1-NP（第二个）
      - InitFC1-Cpl（第三个）
    - 三个 InitFC1 DLLP 必须至少每 34 μs 发送一次
    - 在 Recovery 或 Configuration LTSSM 状态中花费的时间不计入此限制
    - 强烈建议频繁重复 InitFC1 DLLP 传输，特别是当没有其他 TLP 或 DLLP 可供传输时
    - 如果链路上激活了缩放流控，将 InitFC1 DLLP 中的 HdrScale 和 DataScale 字段设置为 01b、10b 或 11b，以指示其在相应 HdrFC 和 DataFC 值上使用的缩放因子
    - 如果发送方不支持缩放流控或链路上未激活缩放流控，将 HdrScale 和 DataScale 字段设置为 00b
    - 除确保 InitFC1 DLLP 传输的必要频率外，数据链路层不得阻止其他传输
    - 注意：这包括所有物理层发起的传输（例如，有序集）、Ack 和 Nak DLLP（如适用），以及使用已完成初始化的 VC 的 TLP（如适用）
    - 处理接收到的 InitFC1 和 InitFC2 DLLP：
      - 记录指示的 HdrFC 和 DataFC 值
      - 如果接收方支持缩放流控，记录指示的 HdrScale 和 DataScale 值
      - 一旦为 VCx 的 P、NP 和 Cpl 各自记录了 FC 单元值，置位标志 FI1
  - 如果满足以下条件，退出到 FC_INIT2：
    - 标志 FI1 已置位，表示已为 VCx 的 P、NP 和 Cpl 各自记录了 FC 单元值

- FC_INIT2 状态的规则：
  - 在 FC_INIT2 状态期间：
    - 事务层必须阻止使用 VCx 的 TLP 传输
    - 按以下相对顺序发送 VCx 的以下三个 InitFC2 DLLP：
      - InitFC2-P（第一个）
      - InitFC2-NP（第二个）
      - InitFC2-Cpl（第三个）
    - 三个 InitFC2 DLLP 必须至少每 34 μs 发送一次
    - 在 Recovery 或 Configuration LTSSM 状态中花费的时间不计入此限制
    - 强烈建议频繁重复 InitFC2 DLLP 传输，特别是当没有其他 TLP 或 DLLP 可供传输时
    - 如果链路上激活了缩放流控，将 InitF2 DLLP 中的 HdrScale 和 DataScale 字段设置为 01b、10b 或 11b，以指示其在相应 HdrFC 和 DataFC 值上使用的缩放因子
    - 如果发送方不支持缩放流控或链路上未激活缩放流控，将 HdrScale 和 DataScale 字段设置为 00b
    - 除确保 InitFC2 DLLP 传输的必要频率外，数据链路层不得阻止其他传输
    - 注意：这包括所有物理层发起的传输（例如，有序集）、Ack 和 Nak DLLP（如适用），以及使用已完成初始化的 VC 的 TLP（如适用）
    - 处理接收到的 InitFC1 和 InitFC2 DLLP：
      - 忽略接收到的 HdrFC、HdrScale、DataFC 和 DataScale 值
      - 收到 VCx 的任何 InitFC2 DLLP 时置位标志 FI2
      - 收到 VCx 上的任何 TLP 或 VCx 的任何 UpdateFC DLLP 时置位标志 FI2
  - 如果满足以下条件，发出完成信号并退出：
    - 标志 FI2 已置位
    - 如果链路上激活了缩放流控，发送方必须在 VCx 的所有 UpdateFC DLLP 中为 HdrScale 和 DataScale 发送 01b、10b 或 11b
    - 如果不支持缩放流控或链路上未激活缩放流控，发送方必须在 VCx 的所有 UpdateFC DLLP 中为 HdrScale 和 DataScale 发送 00b

#### 3.4.2 缩放流控

当可用的流控信用不足以考虑链路往返时间时，链路性能可能会受到影响。这种影响在更高的链路速度下变得更加明显，127 个头部信用和 2047 个数据信用的限制可能会限制性能。缩放流控机制旨在解决此限制。

所有端口都允许支持缩放流控。支持 16.0 GT/s 及更高数据速率的端口必须支持缩放流控。缩放流控激活不影响以 16.0 GT/s 及更高数据速率操作的能力。

当链路未激活缩放流控时，适用以下规则：
- InitFC1、InitFC2 和 UpdateFC DLLP 必须在 HdrScale 和 DataScale 字段中包含 00b
- HdrFC 计数器为 8 位宽，HdrFC DLLP 字段包含计数器的所有位
- DataFC 计数器为 12 位宽，DataFC DLLP 字段包含计数器的所有位

当链路激活缩放流控时，适用以下规则：
- InitFC1 和 InitFC2 DLLP 必须在 HdrScale 字段中包含 01b、10b 或 11b。该值由表 3-2 中定义的指定信用类型的最大未完成头部信用数确定
- InitFC1 和 InitFC2 DLLP 必须在 DataScale 字段中包含 01b、10b 或 11b。该值由表 3-2 中定义的指定信用类型的最大未完成数据载荷信用数确定
- 如果在 FC_INIT1 状态中记录的接收 HdrScale 和 DataScale 值非零，则在此 VC 上启用缩放流控，UpdateFC DLLP 必须在 HdrScale 和 DataScale 字段中包含 01b、10b 或 11b
- 如果在 FC_INIT1 状态中记录的接收 HdrScale 和 DataScale 值为零，则在此 VC 上未启用缩放流控，UpdateFC DLLP 必须在 HdrScale 和 DataScale 字段中包含 00b

**表 3-2 缩放流控缩放因子**

|缩放因子|支持缩放流控|信用类型|最小信用|最大信用|字段宽度|FC DLLP 字段（发送）|FC DLLP 字段（接收）|
|---|---|---|---|---|---|---|---|
|00b|否|Hdr|1|127|8 位|HdrFC|HdrFC|
|00b|否|Data|1|2,047|12 位|DataFC|DataFC|
|01b|是|Hdr|1|127|8 位|HdrFC|HdrFC|
|01b|是|Data|1|2,047|12 位|DataFC|DataFC|
|10b|是|Hdr|4|508|10 位|HdrFC >> 2|HdrFC << 2|
|10b|是|Data|4|8,188|14 位|DataFC >> 2|DataFC << 2|
|11b|是|Hdr|16|2,032|12 位|HdrFC >> 4|HdrFC << 4|
|11b|是|Data|16|32,752|16 位|DataFC >> 4|DataFC << 4|

### 3.6 数据完整性机制

#### 3.6.1 引言

事务层向数据链路层提供 TLP 边界信息。这允许数据链路层为 TLP 应用 TLP 序列号和链路 CRC (LCRC) 用于错误检测。接收数据链路层通过检查 TLP 序列号、LCRC 码以及来自接收物理层的任何错误指示来验证接收到的 TLP。如果 TLP 中存在任何这些错误，则使用数据链路层重试进行恢复。

在支持协议复用的端口上，Symbol +0 的位 7:4 包含非零值的数据包是 PMUX 数据包。对于 TLP，这些位必须是 0000b。详见附录 G。

#### 3.6.2 LCRC、序列号和重试管理（TLP 发送方）

通过数据链路层的 TLP 发送路径（图 3-1 中标记为 1 和 3 的路径）通过应用序列号，然后计算并附加链路 CRC (LCRC) 来准备每个 TLP 的传输，LCRC 用于确保 TLP 在从一个组件到另一个组件的链路传输期间的完整性。TLP 存储在重试缓冲区中，除非收到来自另一组件的无错误接收的肯定确认，否则将重新发送。如果重复传输 TLP 的尝试失败，发送方将确定链路运行不正常，并指示物理层重新训练链路（通过 LTSSM Recovery 状态，第 4.2.6 节）。如果链路重新训练失败，物理层将指示链路不再启动，导致 DLCMSM 移动到 DL_Inactive 状态。

用于确定 TLP LCRC 和序列号以及支持数据链路层重试的机制以概念性的"计数器"和"标志"描述。此描述不暗示也不要求特定的实现，仅用于阐明要求。

#### 3.6.2.1 LCRC 和序列号规则（TLP 发送方）

以下计数器和定时器用于解释本节的其余规则：

- 使用以下 12 位计数器：
  - **NEXT_TRANSMIT_SEQ** - 存储应用于 TLP 的数据包序列号
    - 在 DL_Inactive 状态下设置为 000h
  - **ACKD_SEQ** - 存储最近接收到的 Ack 或 Nak DLLP 中确认的序列号
    - 在 DL_Inactive 状态下设置为 FFFh

- 使用以下 2 位计数器：
  - **REPLAY_NUM** - 计数重试缓冲区已重新传输的次数
    - 在 DL_Inactive 状态下设置为 00b

- 使用以下定时器：
  - **REPLAY_TIMER** - 根据以下规则计算确定何时需要重放的时间：
    - 如果尚未运行，则在任何 TLP 传输或重传的最后一个符号处启动
    - 对于每次重放，当发送第一个要重传的 TLP 的最后一个符号时，重置并重新启动 REPLAY_TIMER
    - 对于每次接收到的 Ack DLLP，如果有更多未确认的 TLP 未完成，且仅当接收到的 Ack DLLP 确认重试缓冲区中的某些 TLP 时，重置并重新启动
    - 注意：这确保 REPLAY_TIMER 仅在进行正向进展时重置
    - 对于每次收到的 Nak（重放期间除外）或当 REPLAY_TIMER 超时时，重置并保持直到满足重新启动条件
    - 在链路重新训练期间不推进（当 LTSSM 处于 Recovery 或 Configuration 状态时保持其值）。参见第 4.2.5.3 节和第 4.2.5.4 节
    - 如果支持协议复用，可选择在接收 PMUX 数据包期间不推进（见附录 G）
    - 当没有未完成的未确认 TLP 时重置并保持

以下规则描述了 TLP 在传递给物理层之前如何准备传输：

- 事务层在传输 TLP 时向数据链路层指示 TLP 的开始和结束
  - 数据链路层将 TLP 视为"黑盒"，不处理或修改 TLP 的内容

- 当从事务层的发送端接受 TLP 时，每个 TLP 被分配一个 12 位序列号
  - 接受来自事务层的 TLP 时，数据包序列号通过以下方式应用于 TLP：
    - 将 NEXT_TRANSMIT_SEQ 中的 12 位值前置到 TLP
    - 在序列号之前将 4 个保留位前置到 TLP（见图 3-15）
  - 如果方程：
    ```
    (NEXT_TRANSMIT_SEQ - ACKD_SEQ) mod 4096 >= 2048    方程 3-1 Tx SEQ Stall
    ```
    为真，则发送方必须停止接受来自事务层的 TLP，直到方程不再为真
  - 在将 NEXT_TRANSMIT_SEQ 应用于从事务层发送端接受的 TLP 后，NEXT_TRANSMIT_SEQ 递增（除非 TLP 被作废）：
    ```
    NEXT_TRANSMIT_SEQ := (NEXT_TRANSMIT_SEQ + 1) mod 4096    方程 3-2 Tx SEQ Update
    ```

- TLP 数据完整性在数据链路层之间传输期间使用 32 位 LCRC 保护
- LCRC 值使用以下机制计算（见图 3-16）：

  - 当物理层使用 128b/130b 编码时，传输 TLP 的所有 DW（见第 4.2.2.3.1 节）
  - 使用计算的 LCRC 值的余数而不反转（通常使用的值的逻辑反）
  - 向发送物理层指示 TLP 已作废
- 当这样做时，发送方不递增 NEXT_TRANSMIT_SEQ

以下规则描述数据链路层重试缓冲区的操作，必要时从中重新传输 TLP：

- 已传输 TLP 的副本必须存储在数据链路层重试缓冲区中，作废的 TLP 除外。

当启动重放时，无论是由于收到 Nak 还是由于 REPLAY_TIMER 超时，必须遵循以下操作序列：

- 如果所有已传输的 TLP 都已被确认（重试缓冲区为空），终止重放，否则继续
- 递增 REPLAY_NUM。当重放由确认重试缓冲区中某些 TLP 的 Nak 接收启动时，REPLAY_NUM 被重置。然后允许（但不要求）递增
  - 如果 REPLAY_NUM 从 11b 溢出到 00b，发送方指示物理层重新训练链路，并等待重新训练完成后再继续重放。这是与端口相关的报告错误（见第 6.2 节）
  - 注意：数据链路层状态，包括重试缓冲区的内容，不会由此操作重置，除非物理层报告 Physical LinkUp = 0b（导致数据链路控制和管理状态机转换到 DL_Inactive 状态）
  - 如果 REPLAY_NUM 未从 11b 溢出到 00b，继续重放
- 阻止接受来自发送事务层的新 TLP
- 完成当前正在传输的任何 TLP 的传输
- 重新传输未确认的 TLP，从最旧的未确认 TLP 开始，按原始传输顺序继续
  - 当发送第一个要重传的 TLP 的最后一个符号时，重置并重新启动 REPLAY_TIMER
  - 一旦所有未确认的 TLP 都已重新传输，返回正常操作
  - 如果在重放期间收到任何 Ack 或 Nak DLLP，发送方允许完成重放而不考虑 Ack 或 Nak DLLP，或跳过任何新确认的 TLP 的重传
  - 一旦发送方开始重新发送 TLP，必须在所有情况下完成该 TLP 的传输
  - 重放期间收到的 Ack 和 Nak DLLP 必须被处理，并可以被合并
    - 示例：如果收到多个 Ack，只需考虑指定最新序列号值的那个 - 指定较早序列号值的 Ack 实际上被"合并"到此 Ack 中
    - 示例：重放期间收到 Nak，然后收到指定较晚序列号的 Ack - Ack 取代 Nak，Nak 被忽略
- 注意：由于重试缓冲区中的所有条目已由发送方的流控门控逻辑在接收方分配了空间，因此不需要进一步的流控同步
- 重新启用接受来自发送事务层的新 TLP

重放可由 REPLAY_TIMER 超时或收到 Nak 启动。以下规则涵盖 REPLAY_TIMER 的超时：

- 如果发送重试缓冲区包含未收到 Ack 或 Nak DLLP 的 TLP，且（由 REPLAY_TIMER 指示）在超过 REPLAY_TIMER 限制的时间内未收到 Ack 或 Nak DLLP，则发送方启动重放
  - 简化的 REPLAY_TIMER 限制为：
    - 当 Extended Synch 位清除时，为 24,000 到 31,000 符号时间的值
    - 当 Extended Synch 位置位时，为 80,000 到 100,000 符号时间的值
    - 如果 Extended Synch 位在有未确认 TLP 未完成时改变状态，允许实现当 Extended Synch 位改变状态或下次重置 REPLAY_TIMER 时调整其 REPLAY_TIMER 限制
  - 支持以所有数据速率操作的 16.0 GT/s 或更高数据速率的实现必须使用简化的 REPLAY_TIMER 限制
  - 仅支持低于 16.0 GT/s 数据速率的实现强烈建议以所有数据速率使用简化的 REPLAY_TIMER 限制，但允许使用 [PCIe-3.1] 中描述的 REPLAY_TIMER 限制

这是重放定时器超时错误，是与端口相关的报告错误（见第 6.2 节）。

**实现说明：确定 REPLAY_TIMER 限制值**

重放主要通过 Nak DLLP 启动，REPLAY_TIMER 作为辅助机制。由于它是辅助机制，REPLAY_TIMER 限制对通过链路传输 TLP 所需的平均时间影响相对较小。简化的 REPLAY_TIMER 限制已定义，因此不需要针对 ASPM L0s、重定时器或本规范先前版本中的其他项目进行调整。

TLP 发送方和合规性测试必须基于在 TLP 发送方端口测量的重放时序。时序从传输 TLP 的最后一个符号或接收 Ack DLLP 的最后一个符号开始，以两者中确定最旧未确认 TLP 的为准。时序在 TLP 重传的第一个符号处结束。

当测量直到 TLP 重传开始的重放时序时，合规性测试必须考虑该方向上已在进行中的任何其他 TLP 或 DLLP 传输（从而阻止 TLP 重传）。

**实现说明：建议的调度传输优先级**

当调度相同类型的多个 DLLP 进行传输但尚未传输时，在许多情况下可以将它们"合并"为单个 DLLP。例如，如果调度的 Ack DLLP 传输因等待另一传输完成而停滞，且在此期间调度了另一个 Ack 进行传输，则只需传输第二个 Ack，因为它提供的信息将取代第一个 Ack 中的信息。

除了来自事务层（或重试缓冲区，如果正在进行重放）的任何 TLP 外，不同类型的多个 DLLP 可能同时调度进行传输，必须按优先级顺序传输。以下列表显示了选择信息进行传输的首选优先级顺序。注意，NOP DLLP 和厂商特定 DLLP 的优先级未列出，因为这些 DLLP 的使用完全取决于实现，没有建议的优先级。注意，此优先级顺序是指导原则，在所有情况下强烈建议使用公平机制，以确保没有任何类型的流量被任何其他类型的流量在扩展或无限期内阻止。注意，Ack 延迟限制值和 REPLAY_TIMER 限制指定了在组件端口测量的要求，组件的内部仲裁策略必须确保满足这些外部测量的要求。

1. 完成任何正在进行的传输（TLP 或 DLLP）（最高优先级）
2. Nak DLLP 传输
3. 由于以下原因调度尽快传输的 Ack DLLP 传输：收到重复 TLP 或 Ack 延迟定时器超时（见第 3.6.3.1 节）
4. 满足第 2.6 节要求的 FC DLLP 传输
5. 重试缓冲区重新传输
6. 来自事务层的 TLP
7. 除满足第 2.6 节要求之外的 FC DLLP 传输
8. 所有其他 DLLP 传输（最低优先级）

#### 3.6.2.2 接收 DLLP 的处理

由于 Ack/Nak 和流控 DLLP 影响在链路上反向流动的 TLP，数据链路层中的 TLP 传输机制还负责处理来自链路上另一组件的 Ack/Nak 和流控 DLLP。这些 DLLP 根据以下规则处理（见图 3-17）：

- 如果物理层指示接收方错误，丢弃当前正在接收的任何 DLLP 并释放为该 DLLP 分配的任何存储空间。注意，向软件报告此类错误由物理层完成（因此不由数据链路层报告）
- 对于所有接收到的 DLLP，通过以下方式检查 CRC 值：
  - 将用于计算已传输 DLLP 的相同算法应用于接收到的 DLLP，不包括接收到的 DLLP 的 16 位 CRC 字段
  - 将计算结果与接收到的 DLLP 的 CRC 字段中的值比较
    - 如果不相等，DLLP 已损坏
  - 损坏的已接收 DLLP 被丢弃。这是 Bad DLLP 错误，是与端口相关的报告错误（见第 6.2 节）
- 未损坏但使用不支持的 DLLP 类型编码的已接收 DLLP 被丢弃，不采取进一步行动。这不被视为错误
- 忽略保留字段中的非零值
- 接收方必须以接收速率处理所有接收到的 DLLP

- 接收到的 NOP DLLP 被丢弃
- 接收到的 FC DLLP 传递给事务层
- 接收到的 PM DLLP 传递给组件的电源管理控制逻辑

- 对于 Ack 和 Nak DLLP，遵循以下步骤（见图 3-18）：
  - 如果 AckNak_Seq_Num 指定的序列号不对应于未确认的 TLP 或 ACKD_SEQ 中的值，DLLP 被丢弃
    - 这是数据链路协议错误，是与端口相关的报告错误（见第 6.2 节）。注意，当没有未完成的未确认 TLP 时（包括重置和第一次 TLP 传输之间的时间）收到 Ack DLLP 不是错误，只要指定的序列号与 ACKD_SEQ 中的值匹配
  - 如果 AckNak_Seq_Num 未指定最近确认的 TLP 的序列号，则 DLLP 确认重试缓冲区中的某些 TLP：
    - 从重试缓冲区中清除从最旧到对应于 AckNak_Seq_Num 的 TLP
    - 用 AckNak_Seq_Num 字段中的值加载 ACKD_SEQ
    - 重置 REPLAY_NUM 和 REPLAY_TIMER
  - 如果 DLLP 是 Nak，启动重放（见上文）

注意：收到 Nak 不是报告错误。

以下规则描述数据链路层重试缓冲区的操作，必要时从中重新传输 TLP：

- 已传输 TLP 的副本必须存储在数据链路层重试缓冲区中

#### 3.6.3 LCRC 和序列号（TLP 接收方）

通过数据链路层的 TLP 接收路径（图 3-1 中标记为 2 和 4 的路径）通过检查 LCRC 和序列号来处理物理层接收的 TLP，如果正确则将 TLP 传递给接收事务层，如果损坏则请求重放。

用于检查 TLP LCRC 和序列号以及支持数据链路层重试的机制以概念性的"计数器"和"标志"描述。此描述不暗示或要求特定的实现，仅用于阐明要求。

#### 3.6.3.1 LCRC 和序列号规则（TLP 接收方）

以下计数器、标志和定时器用于解释本节的其余规则：

- 使用以下 12 位计数器：
  - **NEXT_RCV_SEQ** - 存储下一个 TLP 的预期序列号
    - 在 DL_Inactive 状态下设置为 000h

- 使用以下标志：
  - **NAK_SCHEDULED**
    - 在 DL_Inactive 状态下清除

- 使用以下定时器：
  - **AckNak_LATENCY_TIMER** - 根据以下规则计算确定何时调度发送 Ack DLLP 的时间：
    - 在 DL_Inactive 状态下设置为 0
    - 每次调度发送 Ack 或 Nak DLLP 时从 0 重新启动；当所有接收到的 TLP 都已通过发送 Ack DLLP 确认时，重置为 0
    - 如果最初没有未确认的 TLP，然后收到一个 TLP，则 AckNak_LATENCY_TIMER 仅在 TLP 已转发到接收事务层后才开始计数

以下规则按顺序应用，描述如何处理接收到的 TLP 以及什么事件触发 Ack 和 Nak DLLP 的传输（见图 3-19）：

- 如果物理层指示接收方错误，丢弃当前正在接收的任何 TLP 并释放为该 TLP 分配的任何存储空间。注意，向软件报告此类错误由物理层完成（因此不由数据链路层报告）
  - 如果在指示接收方错误时正在接收 TLP 且 NAK_SCHEDULED 标志清除：
    - 立即调度发送 Nak DLLP
    - 置位 NAK_SCHEDULED 标志

- 如果物理层报告接收到的 TLP 已作废，且 LCRC 是计算值的逻辑非，则丢弃 TLP 并释放为该 TLP 分配的任何存储空间。这不被视为错误
- 如果 TLP 已作废但 LCRC 与计算值的逻辑非不匹配，TLP 已损坏，丢弃 TLP 并释放为该 TLP 分配的任何存储空间
  - 如果 NAK_SCHEDULED 标志清除：
    - 立即调度发送 Nak DLLP
    - 置位 NAK_SCHEDULED 标志
  - 这是 Bad TLP 错误，是与端口相关的报告错误（见第 6.2 节）

- 通过以下方式检查 LCRC 值：
  - 将用于计算（上述）的相同算法应用于接收到的 TLP，不包括接收到的 TLP 的 32 位 LCRC 字段
  - 将计算结果与接收到的 TLP 的 LCRC 字段中的值比较
    - 如果不相等，TLP 已损坏 - 丢弃 TLP 并释放为该 TLP 分配的任何存储空间
    - 如果 NAK_SCHEDULED 标志清除：
      - 立即调度发送 Nak DLLP
      - 置位 NAK_SCHEDULED 标志
    - 这是 Bad TLP 错误，是与端口相关的报告错误（见第 6.2 节）

- 如果 TLP 序列号不等于存储在 NEXT_RCV_SEQ 中的预期值：
  - 丢弃 TLP 并释放为该 TLP 分配的任何存储空间
  - 如果 TLP 序列号满足以下方程：
    ```
    (NEXT_RCV_SEQ - TLP Sequence Number) mod 4096 <= 2048
    ```
    TLP 是重复的，调度发送 Ack DLLP（按传输优先级规则）
  - 否则，TLP 失序（表示一个或多个 TLP 丢失）：
    - 如果 NAK_SCHEDULED 标志清除：
      - 立即调度发送 Nak DLLP
      - 置位 NAK_SCHEDULED 标志
    - 这是 Bad TLP 错误，是与端口相关的报告错误（见第 6.2 节）。无论 NAK_SCHEDULED 标志的状态如何，都允许这是与端口相关的报告错误（见第 6.2 节），图 3-17 中说明了此允许行为。但是，为了防止错误污染，建议端口仅当 NAK_SCHEDULED 标志清除时报告此类错误

  - 移除四个保留位、TLP 序列号和 LCRC（见图 3-14），并将 TLP 的其余部分转发给接收事务层
    - 数据链路层在传输 TLP 时向事务层指示 TLP 的开始和结束
    - 数据链路层将 TLP 视为"黑盒"，不处理或修改 TLP 的内容

  - 用于 Ack DLLP 传输的链路已处于 L0 或已转换到 L0
  - 注意：如果尚未处于 L0，链路必须转换到 L0 以传输 Ack DLLP
  - 链路上当前未传输其他 TLP 或 DLLP
  - NAK_SCHEDULED 标志清除

**实现说明：重试缓冲区大小确定**

重试缓冲区应足够大，以确保在正常操作条件下，传输永远不会因重试缓冲区已满而受到限制。在确定最佳缓冲区大小时，必须考虑 Ack 延迟值、接收方已在传输另一个 TLP 引起的 Ack 延迟、物理链路互连引起的延迟以及处理接收到的 Ack DLLP 所需的时间。

给定两个组件 A 和 B，A 的接收方所需的 L0s 退出延迟应在确定 A 的发送重试缓冲区大小时考虑，如下例所示：

- A 在其到 B 的发送路径上退出 L0s 并开始向 B 发送长突发写请求
- B 在其到 A 的发送路径上启动 L0s 退出，但 A 的接收方所需的 L0s 退出时间很大
- 同时，B 无法向 A 发送 Ack DLLP，A 因重试缓冲区空间不足而停顿
- 从 B 到 A 的发送路径返回 L0，B 向 A 发送 Ack DLLP，停顿解决

可以通过将组件的发送方重试缓冲区大小与组件接收方所需的 L0s 退出延迟相匹配来避免此停顿，或者相反，将接收方 L0s 退出延迟与重试缓冲区的所需大小相匹配。

选择 Ack 延迟限制值是为了允许实现获得良好性能，而无需过大且不经济的重试缓冲区。为了在具有不同实现和应用通用互连中实现一致的性能，必须为所有组件设置相同的要求，而不考虑任何特定组件的应用空间。如果组件不需要链路的全部传输带宽，它可以将其重试缓冲区大小减小到低于在指定 Ack 延迟限制值下保持可用重试缓冲区空间所需的最小大小。

注意，指定的 Ack 延迟限制值确保允许的未完成 TLP 序列号范围永远不会成为导致传输停顿的限制因素。

重定时器会增加延迟（见第 4.3.8 节），在 SRIS 中操作会增加延迟。强烈建议实现在确定最佳缓冲区大小时考虑这些影响。

## 第四章 物理层逻辑块

### 4.1 引言

物理层将事务层和数据链路层与用于链路数据交换的信令技术隔离。物理层分为逻辑子块和电子子块（见图 4-1）。

第 4 章描述逻辑子块，第 8 章描述电子子块。

### 4.2 逻辑子块

逻辑子块有两个主要部分：发送部分，准备从数据链路层传出的信息以供电子子块传输；接收部分，识别并准备接收到的信息后传递给数据链路层。

逻辑子块和电子子块通过状态和控制寄存器接口或功能等效物协调每个收发器的状态。逻辑子块指导物理层的控制和管理功能。

当数据速率为 2.5 GT/s 或 5.0 GT/s 时，PCI Express 使用 8b/10b 编码。对于大于或等于 8.0 GT/s 的数据速率，它使用逐通道编码以及物理层封装。

#### 4.2.1 2.5 GT/s 和 5.0 GT/s 数据速率的编码

##### 4.2.1.1 符号编码

在 2.5 和 5.0 GT/s 时，PCI Express 使用 8b/10b 传输编码。此传输编码的定义与 ANSI X3.230-1994 第 11 条（以及 IEEE 802.3z 36.2.4）中指定的相同。使用此方案，8 位数据字符被视为 3 位和 5 位，分别映射到 4 位码组和 6 位码组。控制位与数据字符一起用于识别何时编码 8b/10b 传输编码中包含的 12 个特殊符号之一。这些码组连接形成 10 位符号。如图 4-2 所示，ABCDE 映射到 abcdei，FGH 映射到 fghj。

发送方允许在处于电气空闲状态后首次发送差分数据时选择任何极性，除非另有要求。然后发送方必须遵循正确的 8b/10b 编码规则，直到进入下一个电气空闲状态。

检测到电气空闲退出的接收方的初始极性设置为用于获得符号锁定的第一个符号的极性。如果由于实现特定的错误数量而在差分信息传输期间丢失并重新获得符号锁定，也可以重新初始化极性。设置初始极性后接收到的所有后续符号必须在对应于当前运行极性的正确列中找到。

如果接收到的符号在对应于不正确运行极性的列中找到，或者符号不对应于任何一列，物理层必须通知数据链路层接收到的符号无效。这是接收方错误，是与端口相关的报告错误（见第 6.2 节）。

##### 4.2.1.2 符号在通道上的成帧和应用

符号在通道上的成帧和应用有两类。第一类由有序集组成，第二类由 TLP 和 DLLP 组成。有序集总是在每个通道上串行传输，使得完整的有序集同时出现在多通道链路的所有通道上。

成帧机制使用特殊符号 K28.2 "SDP" 启动 DLLP，使用特殊符号 K27.7 "STP" 启动 TLP。特殊符号 K29.7 "END" 用于标记 TLP 或 DLLP 的结束。

概念性的符号流必须从其内部表示（取决于实现）映射到外部通道。符号映射到通道上，使得第一个符号（表示字符 0）放置在通道 0 上；第二个放置在通道 1 上；以此类推。x1 链路表示一种退化情况，映射是平凡的，所有符号按顺序放置在单个通道上。

当没有数据包信息或特殊有序集正在传输时，发送方处于逻辑空闲状态。在此期间，必须传输空闲数据。空闲数据必须由数据字节 0（十六进制 00）组成，按照第 4.2.1.3 节的规则进行加扰，并按照第 4.2.1.1 节的规则进行 8b/10b 编码，方式与 TLP 和 DLLP 数据符号的加扰和编码相同。同样，当接收方没有接收任何数据包信息或特殊有序集时，接收方处于逻辑空闲状态，必须接收如上所述的空闲数据。在空闲数据传输期间，必须继续按照第 4.2.7 节的规定传输 SKP 有序集。

对于以下规则，"放置"定义为发送方将符号放入链路正确通道的要求。

- TLP 必须通过在 TLP 开始处放置 STP 符号，在 TLP 结束处放置 END 符号或 EDB 符号来进行成帧（见图 4-5）
- 正确形成的 TLP 在 STP 和 END 或 EDB 符号之间至少包含 18 个符号。如果接收到的序列在 STP 和 END 或 EDB 符号之间的符号少于 18 个，接收方允许将其视为接收方错误
  - 如果检查，这是与接收端口相关的报告错误（见第 6.2 节）
- DLLP 必须通过在 DLLP 开始处放置 SDP 符号，在 DLLP 结束处放置 END 符号来进行成帧（见图 4-6）
- 逻辑空闲定义为没有信息：TLP、DLLP 或任何类型的特殊符号正在传输/接收的一个或多个符号时间期间。与电气空闲不同，在逻辑空闲期间，正在传输和接收空闲符号 (00h)
  - 当发送方处于逻辑空闲时，必须在所有通道上传输逻辑空闲数据 (00h)。这按照第 4.2.1.3 节的规则进行加扰
  - 接收方必须忽略传入的逻辑空闲数据，除了加扰序列外，不得对任何特定数据模式有任何依赖
- 对于宽于 x1 的链路，当从逻辑空闲链路条件开始传输 TLP 时，STP 符号（表示 TLP 的开始）必须放置在通道 0 上
- 对于宽于 x1 的链路，当从逻辑空闲链路条件开始传输 DLLP 时，SDP 符号（表示 DLLP 的开始）必须放置在通道 0 上
- STP 符号在每个符号时间内在链路上的放置频率不得超过一次
- SDP 符号在每个符号时间内在链路上的放置频率不得超过一次
- 只要满足上述规则，TLP 和 DLLP 传输允许连续跟随
- 一个 STP 符号和一个 SDP 符号可以在同一符号时间内放置在链路上
  - 宽于 x4 的链路可以在通道 4*N 上放置 STP 和 SDP 符号，其中 N 为正整数。例如，对于 x8，STP 和 SDP 符号可以放置在通道 0 和 4 上；对于 x16，STP 和 SDP 符号可以放置在通道 0、4、8 或 12 上
- 对于 N 为 8 或更大的 xN 链路，如果 END 或 EDB 符号放置在通道 K 中（其中 K 不等于 N-1），且通道 K+1 中没有紧随其后的 STP 或 SDP 符号（即没有立即跟随的 TLP 或 DLLP），则必须在通道 K+1 到通道 N-1 中放置 PAD 符号
  - 例如，在 x8 链路上，如果 END 或 EDB 放置在通道 3 中，当后面没有 STP 或 SDP 时，必须在通道 4 到 7 中放置 PAD
- EDB 符号用于标记已作废 TLP 的结束。有关 EDB 使用的信息，请参阅第 3.6.2.1 节
- 接收方可选择检查本节规则的违规情况。这些检查是独立可选的（见第 6.2.3.4 节）。如果检查，违规是接收方错误，是与端口相关的报告错误（见第 6.2 节）

##### 4.2.1.3 数据加扰

为了改善链路的电气特性，通常对数据进行加扰。这涉及将数据流与线性反馈移位寄存器 (LFSR) 生成的模式进行异或操作。在发送端，加扰应用于 8b/10b 编码之前的字符。在接收端，解扰应用于 8b/10b 解码之后的字符。

在多通道链路上，加扰功能可以用一个或多个 LFSR 实现。当每条链路有多个发送 LFSR 时，它们必须协同工作，在每个 LFSR 中保持相同的同步（通道间输出偏斜）值。当每条链路有多个接收 LFSR 时，它们必须协同工作，在每个 LFSR 中保持相同的同步（通道间偏斜）值。无论它们如何实现，LFSR 必须以逐通道为基础与数据交互，就像该链路内的每个通道都有一个独立的 LFSR，如此处所述。

LFSR 如图 4-10 所示。加扰或解扰通过将 8 位 (D0-D7) 字符与 LFSR 的 16 位 (D0-D15) 输出串行异或来执行。LFSR 的输出 D15 与要处理的数据 D0 异或。然后 LFSR 和数据寄存器串行推进，对 D1 到 D7 重复输出处理。LFSR 在数据异或后推进。LFSR 实现多项式：

```
G(X) = X^16 + X^5 + X^4 + X^3 + 1
```

数据链路层用于通知物理层禁用加扰的机制和/或接口取决于实现，超出了本规范的范围。数据加扰规则如下：

- COM 符号初始化 LFSR
- LFSR 值对每个符号（SKP 除外）推进八次串行移位
- 所有数据符号（D 码）都被加扰，但有序集内（例如 TS1、TS2、EIEOS）、合规模式（见第 4.2.8 节）和修改后的合规模式（见第 4.2.9 节）中的符号除外
- 所有特殊符号（K 码）都不加扰
- LFSR 种子 (D0-D15) 的初始值为 FFFFh。COM 退出发送 LFSR 后立即初始化发送端的 LFSR。每当 COM 进入该链路任何通道上的接收 LFSR 时，接收端的 LFSR 被初始化
- 加扰只能在配置结束时禁用（见第 4.2.6.3.5 节）
- 加扰不适用于环回从设备
- 加扰在检测中默认始终启用

**实现说明：禁用加扰**

禁用加扰旨在帮助简化和测试调试设备。在测试和调试环境中，控制确切的数据模式很有用。由于加扰在物理层重置，因此没有合理的方法通过软件可靠地控制数据转换的状态。因此，TS1 和 TS2 有序集中的禁用加扰位用于这些目的。

数据链路层用于通知物理层禁用加扰的机制和/或接口取决于实现，超出了本规范的范围。

有关加扰的更多信息，请参阅附录 C。

#### 4.2.2 8.0 GT/s 及更高数据速率的编码

当 PCI Express 链路以 8.0 GT/s 或更高的数据速率运行时，它使用本小节描述的编码规则：128b/130b 编码。为了向后兼容，链路最初使用 8b/10b 编码以 2.5 GT/s 数据速率训练到 L0，如第 4.2.1 节所述，然后当数据速率更改为 8.0 GT/s 或更高时，使用 128b/130b 编码。128b/130b 编码是一种链路范围的数据包化机制和带加扰的逐通道块编码。数据传输的基本实体是 8 位数据字符，称为符号，如图 4-11 和图 4-12 所示。

**实现说明：128b/130b 编码方案中的符号**

在 128b/130b 编码方案中，符号长度为一个字节，类似于 8b/10b 编码的 10 位符号。

##### 4.2.2.1 通道级编码

物理层使用逐通道块编码。每个块由 2 位同步头和载荷组成。有两个有效的同步头编码：10b 和 01b。同步头定义块包含的载荷类型。

同步头为 10b 表示数据块。每个数据块有 128 位载荷，导致块大小为 130 位。载荷是第 4.2.2.3 节中描述的数据流。

同步头为 01b 表示有序集块。每个有序集块有 128 位载荷，导致块大小为 130 位，SKP 有序集除外，它可以是可变长度。

多通道链路的所有通道必须同时传输具有相同同步头的块，但在 Polling.Compliance 中传输抖动测量模式时除外。

位传输顺序如下。表示为 'H1H0' 的同步头放置在通道上，从 'H0' 开始，以 'H1' 结束。表示为 'S7S6S5S4S3S2S1S0' 的符号放置在通道上，从 'S0' 开始，以 'S7' 结束。在显示时间刻度的图中，位表示传输顺序。在布局图中，位按小端格式排列，与本规范其他章节中的数据包布局图一致。

##### 4.2.2.2 有序集块

有序集块包含同步头，后跟一个有序集。多通道链路的所有通道必须同时传输相同的有序集类型。有序集的第一个符号定义有序集的类型。有序集的后续符号由有序集类型定义，在多通道链路的通道间不必相同。有序集在第 4.2.4 节和第 4.2.7 节中详细描述。

###### 4.2.2.2.1 块对齐

在链路训练期间，电气空闲退出有序集 (EIEOS) 的 130 位是唯一的位模式，接收方用于确定接收位流中块同步头的位置。概念上，接收方可以处于块对齐的三个不同阶段：未对齐、已对齐和已锁定。这些阶段用于说明所需行为，但不意味着指定所需的实现。

**未对齐阶段**：接收方在电气空闲期后进入此阶段，例如当数据速率更改为使用 128b/130b 编码的速率时，或当它们退出低功率链路状态时，或被指示时（通过实现特定的方法）。在此阶段，接收方监视接收位流中的 EIEOS 位模式。当检测到一个时，它们调整对齐到它，并进入已对齐阶段。

**已对齐阶段**：接收方监视接收位流中的 EIEOS 位模式和接收块中的数据流起始 (SDS) 有序集。如果在当前对齐不匹配的对齐上检测到 EIEOS 位模式，接收方必须调整其对齐到新接收到的 EIEOS 位模式。如果接收到 SDS 有序集，接收方进入已锁定阶段。如果接收到未定义的同步头（00b 或 11b），接收方允许返回到未对齐阶段。

**已锁定阶段**：接收方在此阶段不得调整其块对齐。SDS 有序集后预期接收数据块，调整块对齐会干扰这些块的处理。如果接收到未定义的同步头，接收方必须返回到未对齐或已对齐阶段。

**实现说明：检测块对齐丢失**

在训练序列期间传输的 EIEOS 和 TS 有序集序列将导致未对齐的接收方检测到未定义的同步头。

**附加要求：**
- 在已对齐或已锁定阶段时，接收方在收到 SKP 有序集时必须根据需要调整其对齐。有关 SKP 有序集的更多信息，请参阅第 4.2.7 节
- 在任何 LTSSM 转换到 Recovery 后，接收方必须忽略所有接收到的 TS 有序集，直到收到 EIEOS。概念上，接收 EIEOS 验证接收方的对齐，并允许 TS 有序集处理继续进行。如果接收到的 EIEOS 启动从 L0 到 Recovery 的 LTSSM 转换，接收方允许处理 EIEOS 后的任何 TS 有序集，或在进入 Recovery 后收到另一个 EIEOS 之前忽略它们
- 只要数据流处理已停止，接收方允许从已锁定阶段转换到未对齐或已对齐阶段。有关数据流要求的更多信息，请参阅第 4.2.2.3 节
- 环回主设备：在 Loopback.Entry 期间，主设备必须能够将其接收方的块对齐调整到接收到的 EIEOS 位模式。在 Loopback.Active 期间，主设备允许传输 EIEOS 并将其接收方的块对齐调整到环回位流
- 环回从设备：在 Loopback.Entry 期间，从设备必须能够将其接收方的块对齐调整到接收到的 EIEOS 位模式。在 Loopback.Active 期间，从设备不得调整其接收方的块对齐。概念上，当从设备开始环回接收到的位流时，接收方被引导到已锁定阶段

##### 4.2.2.3 数据块

数据块的载荷是由成帧令牌、TLP 和 DLLP 定义的符号流，称为"数据流"。数据流的每个符号放置在链路的单个通道上，符号流分布在链路的所有通道上并跨越块边界。

数据流从 SDS 有序集后的数据块的第一个符号开始。它在检测到成帧错误时结束，或在与 SKP 有序集以外的有序集之前的数据块的最后一个符号处结束。数据流中出现的 SKP 有序集有特定要求，如以下章节所述。

###### 4.2.2.3.1 成帧令牌

物理层使用的成帧令牌如表 4-2 所示。每个成帧令牌指定或暗示与令牌关联的符号数量，因此下一个成帧令牌的位置。图 4-15 显示了在 x8 链路上传输 TLP、DLLP 和 IDL 的示例。

数据流的第一个成帧令牌始终位于数据流第一个数据块的通道 0 的符号 0 中。在本章其余部分，成帧令牌和令牌这两个术语可互换使用。

**表 4-2 成帧令牌编码**

|成帧令牌类型|描述|
|---|---|
|IDL|逻辑空闲。成帧令牌为 1 个符号。当没有传输 TLP 或 DLLP 或其他成帧令牌时传输此令牌|
|SDP|DLLP 起始。成帧令牌为 2 个符号长，后跟 DLLP 信息|

物理层 DLLP 布局如图 4-14 所示。符号 0 和 1 是 SDP 令牌，符号 2 到 7 是数据链路层 DLLP 信息。

物理层 TLP 布局如图 4-14 所示。STP 成帧令牌的详细信息如图 4-13 所示。正在传输的 TLP 长度（以 DW 为单位）由名为 TLP Length 的 11 位字段指定。TLP Length 字段是传输的总信息量，包括成帧令牌、TLP 前缀（如有）、TLP 头部、TLP 数据载荷（如有）、TLP 摘要（如有）和 TLP LCRC。例如，如果 TLP 有 3 DW 头部、1 DW 数据载荷且不包含 TLP 摘要，则 TLP Length 字段值为 6：1（成帧令牌）+ 0（TLP 前缀）+ 3（TLP 头部）+ 1（TLP 数据载荷）+ 0（TLP 摘要）+ 1（TLP LCRC）。如果同一 TLP 包含 TLP 摘要，TLP Length 字段值将为 7。当 TLP 被作废时，EDB 令牌被视为 TLP 的扩展，但不计入 TLP Length 字段的计算。

TLP Length 字段由 4 位 CRC（帧 CRC）保护，偶校验位（帧奇偶校验）保护 TLP Length 和帧 CRC 字段。帧 CRC 和帧奇偶校验计算如下：

```
C[0] = L[10] ^ L[7] ^ L[6] ^ L[4] ^ L[2] ^ L[1] ^ L[0]
C[1] = L[10] ^ L[9] ^ L[7] ^ L[5] ^ L[4] ^ L[3] ^ L[2]
C[2] = L[9] ^ L[8] ^ L[6] ^ L[4] ^ L[3] ^ L[2] ^ L[1]
C[3] = L[8] ^ L[7] ^ L[5] ^ L[3] ^ L[2] ^ L[1] ^ L[0]
P = L[10] ^ L[9] ^ L[8] ^ L[7] ^ L[6] ^ L[5] ^ L[4] ^ L[3] ^ L[2] ^ L[1] ^ L[0] ^ C[3] ^ C[2] ^ C[1] ^ C[0]
```

帧奇偶校验简化为：
```
P = L[10] ^ L[9] ^ L[8] ^ L[6] ^ L[5] ^ L[2] ^ L[0]
```

在上述方程中，TLP Length 字段表示为 L[10:0]，其中 L[0] 是最低有效位，L[10] 是最高有效位。发送方在传输前计算帧 CRC 和帧奇偶校验。接收方必须使用与发送方相同的算法计算帧 CRC 和帧奇偶校验，然后将计算值与接收值进行比较。

STP 令牌的 TLP Length 字段值不为 1。如果接收到的符号序列与 TLP Length 字段值为 1 的 STP 令牌格式匹配，则评估这些符号以确定它们是否与 EDS 令牌匹配。

**实现说明：帧 CRC 和帧奇偶校验**

帧 CRC 位有效地计算为 (L[0] X^14 + L[1] X^13 + ... + L[9] X^5 + L[10] X^4) mod (X^4 + X + 1)。注意，X^4 + X + 1 是本原多项式，CRC 可以检测两位错误。帧奇偶校验位可以检测奇数位错误。因此，帧 CRC 和帧奇偶校验一起保证对 TLP Length 字段的三位错误检测。注意，尽管在简化的帧奇偶校验方程中并非所有项都存在，但它仍然保持检测奇数位错误的属性。只有那些在偶数个 CRC 项中出现的 TLP Length 字段位才用于计算。

注意，对于 TLP，数据链路层在计算 LCRC 之前将 4 个保留位 (0000b) 前置到 TLP 序列号字段。当使用 128b/130b 编码时，这些保留位不显式传输，接收方在计算 LCRC 时假设接收到的 4 位为 0000b。

包含大于 1535 的 TLP Length 字段的数据包是 PMUX 数据包。对于此类数据包，实际数据包长度以不同方式计算，STP 令牌中的 TLP 序列号字段包含其他信息，链路 CRC 使用不同规则计算。详见附录 G。

包含 1152 到 1535（含）之间 TLP Length 字段的数据包保留用于未来标准化。

发送方必须传输 STP 成帧令牌的 TLP Length 字段指定的 TLP 的所有 DW。使用 128b/130b 编码时 TLP 永远不会被截断——即使被作废。图 4-16 显示了已作废的 23 DW TLP 的示例。

图 4-17 显示了 TLP、DLLP、IDL 和 EDS 令牌后跟 SKP 有序集的示例。SKP 有序集在第 4.2.7.2 节中定义。

###### 4.2.2.3.2 发送方成帧要求

以下要求适用于传输的数据流。

**传输 TLP：**
- 立即传输 STP 令牌，后跟数据链路层提供的完整 TLP 信息
- 必须传输 STP 令牌的 TLP Length 字段指定的 TLP 的所有 DW，即使 TLP 已作废
- 如果 TLP 已作废，必须立即在 TLP 之后传输 EDB 令牌。TLP 的最后一个符号和 EDB 令牌的第一个符号之间不得有符号。已作废 TLP 的 STP 令牌的 TLP Length 字段值不会因 EDB 令牌而调整
- STP 令牌在每个符号时间内的传输频率不得超过一次

**传输 DLLP：**
- 立即传输 SDP 令牌，后跟数据链路层提供的完整 DLLP 信息
- 必须传输 DLLP 的所有 6 个符号
- SDP 令牌在每个符号时间内的传输频率不得超过一次

**在数据流内传输 SKP 有序集：**
- 在当前数据块的最后一个 DW 中传输 EDS 令牌。例如，对于 x1 链路，令牌在符号时间 12-15 的通道 0 上传输；对于 x16 链路，在符号时间 15 的通道 12-15 上传输
- 在当前数据块之后传输 SKP 有序集
- 在 SKP 有序集之后传输数据块。数据流从数据块的第一个符号恢复。如果调度传输多个 SKP 有序集，每个 SKP 有序集必须前面有一个带有 EDS 令牌的数据块

**结束数据流：**
- 在当前数据块的最后一个 DW 中传输 EDS 令牌，随后在下一个块中传输 EIOS 或 EIEOS。EIOS 用于 LTSSM 电源管理状态转换，EIEOS 用于所有其他情况。例如，对于 x1 链路，令牌在符号时间 12-15 的通道 0 上传输；对于 x16 链路，在符号时间 15 的通道 12-15 上传输

- 当不传输 TLP、DLLP 或其他成帧令牌时，必须在所有通道上传输 IDL 令牌

**多通道链路：**
- 传输 IDL 令牌后，下一个 STP 或 SDP 令牌的第一个符号必须在将来符号时间的通道 0 中传输。EDS 令牌可以在同一符号时间内的 IDL 令牌之后传输，因为它必须在块的最后一个 DW 中传输
- 对于 N 为 8 或更大的 xN 链路，如果 EDB 令牌、TLP 或 DLLP 在通道 K 中结束（其中 K 不等于 N-1），且通道 K+1 中没有紧随其后的 STP、SDP 或 EDB 令牌的第一个符号，则必须在通道 K+1 到 N-1 中放置 IDL 令牌。例如，在 x8 链路上，如果 TLP 或 DLLP 在通道 3 中结束，必须在通道 4 到 7 中放置 IDL 令牌。EDS 令牌是此要求的例外，可以在 IDL 令牌之后传输
- 令牌、TLP 和 DLLP 允许连续跟随，使得可以在同一符号时间内传输多个令牌，只要其传输符合本节所述的其他要求
  - 宽于 x4 的链路可以在通道 4*N 上开始放置令牌，其中 N 为正整数。例如，令牌可以放置在 x8 链路的通道 0 和 4 中，令牌可以放置在 x16 链路的通道 0、4、8 或 12 中

###### 4.2.2.3.3 接收方成帧要求

以下要求适用于接收的数据流和数据流开始和结束时发生的块类型转换。

- 当处理预期为成帧令牌的符号时，接收与成帧令牌定义不匹配的符号或符号序列是成帧错误。强烈建议多通道链路的接收方在通道错误状态寄存器中为接收到预期成帧令牌的第一个符号且该符号与 STP（仅位 [3:0]）、IDL、SDP、EDB 或 EDS 令牌的符号 +0 不匹配的通道报告错误（见图 4-13）
- 本节中的所有可选错误检查和错误报告都是独立可选的（见第 6.2.3.4 节）

**收到 STP 令牌时：**
- 接收方必须计算接收到的 TLP Length 字段的帧 CRC 和帧奇偶校验，并将结果与接收到的帧 CRC 和帧奇偶校验字段进行比较。帧 CRC 或帧奇偶校验不匹配是成帧错误
  - 具有成帧错误的 STP 令牌在向数据链路层报告时不被视为 TLP 的一部分
- 如果 TLP Length 字段为 1，符号不是 STP 令牌，而是评估以确定它们是否为 EDS 令牌
- 接收方允许检查 TLP Length 字段值是否为 0。如果检查，接收 TLP Length 字段值为 0 是成帧错误
- 接收方允许检查 TLP Length 字段值是否为 2、3 或 4。如果检查，接收此类 TLP Length 字段是成帧错误
- 接收方允许检查 TLP Length 字段值是否在 1152 到 1535（含）之间。如果检查，接收此类 TLP Length 字段是成帧错误
- 不支持协议复用的端口上的接收方允许检查 TLP Length 字段值是否大于 1535。如果检查，接收此类 TLP Length 字段是成帧错误
- 支持协议复用的端口上的接收方必须将 TLP Length 字段大于 1535 的 STP 令牌作为 PMUX 数据包的开始处理，如附录 G 所定义
- 下一个要处理的令牌从紧随 TLP 最后一个 DW 的符号开始，由 TLP Length 字段确定
  - 接收方必须评估此符号以确定它是否是 EDB 令牌的第一个符号，从而确定 TLP 是否已作废。参见 EDB 令牌要求
- 接收方允许检查在单个符号时间内是否收到多个 STP 令牌。如果检查，在单个符号时间内接收多个 STP 令牌是成帧错误

**收到 EDB 令牌时：**
- 如果在 TLP 之后立即收到 EDB 令牌（TLP 的最后一个符号和 EDB 令牌的第一个符号之间没有符号），接收方必须通知数据链路层已收到 EDB 令牌。接收方允许在处理 EDB 令牌的第一个符号后或处理 EDB 令牌的任何或所有剩余符号后通知数据链路层已收到 EDB 令牌。无论何时通知数据链路层收到 EDB 令牌，接收方必须检查 EDB 令牌的所有符号。接收到与 EDB 令牌定义不匹配的符号是成帧错误
- 在 TLP 之后以外的任何时间接收 EDB 令牌是成帧错误
- 下一个要处理的令牌从紧随 EDB 令牌的符号开始

**在链路上数据块最后四个符号中收到 EDS 令牌时：**
- 接收方必须停止处理数据流
- 在 EDS 令牌后面的块中接收除 SKP、EIOS 或 EIEOS 以外的有序集是成帧错误
- 如果在 EDS 令牌后面的块中收到 SKP 有序集，接收方从紧随 SKP 有序集的数据块的第一个符号恢复数据流处理，除非检测到成帧错误

**收到 SDP 令牌时：**
- 下一个要处理的令牌从紧随 DLLP 最后一个符号的符号开始
- 接收方允许检查在单个符号时间内是否收到多个 SDP 令牌。如果检查，在单个符号时间内接收多个 SDP 令牌是成帧错误

**收到 IDL 令牌时：**
- 对于 x1 链路，下一个要处理的令牌从接收到的下一个符号开始
- 对于 x2 链路，下一个要处理的令牌从下一个符号时间的通道 0 中接收到的符号开始。强烈建议接收方检查在通道 0 中收到 IDL 令牌后通道 1 中收到的符号（如果它没有收到 IDL）是否也是 IDL，并在通道错误状态寄存器中为通道 1 报告错误。如果检查，接收非 IDL 符号是成帧错误
- 对于 x4 链路，下一个要处理的令牌从下一个符号时间的通道 0 中接收到的符号开始。强烈建议接收方检查在通道 0 中收到 IDL 令牌后通道 1-3 中收到的符号是否也是 IDL，并在通道错误状态寄存器中为没有收到 IDL 的通道报告错误。如果检查，接收非 IDL 符号是成帧错误
- 对于 x8、x12、x16 和 x32 链路，下一个要处理的令牌从 IDL 令牌后下一个 DW 对齐通道中接收到的符号开始。例如，如果在 x16 链路的通道 4 中收到 IDL 令牌，下一个令牌位置从同一符号时间的通道 8 开始。但是，如果在 x8 链路的通道 4 中收到 IDL 令牌，下一个令牌位置从下一个符号时间的通道 0 开始。强烈建议接收方检查 IDL 令牌和下一个令牌位置之间收到的符号是否也是 IDL，并在通道错误状态寄存器中为没有收到 IDL 的通道报告错误。如果检查，接收非 IDL 符号是成帧错误。注意：预期在同一符号时间内在 IDL 令牌之后接收到的唯一令牌是额外的 IDL 令牌或 EDS 令牌

**处理数据流时，接收方还必须检查每个通道接收的块类型（在考虑通道间去偏斜后）是否符合以下条件：**
- 在 SDS 有序集之后立即在任何通道上接收有序集块是成帧错误
- 接收具有未定义块类型（同步头为 11b 或 00b）的块是成帧错误。强烈建议多通道链路的接收方在通道错误状态寄存器中为收到未定义块类型的任何通道报告错误
- 在没有在前面的块中收到 EDS 令牌的情况下在任何通道上接收有序集块是成帧错误。例如，在没有前面 EDS 令牌的情况下接收 SKP 有序集是成帧错误。此外，在数据流内立即跟随另一个有序集块（包括另一个 SKP 有序集）的 SKP 有序集是成帧错误。强烈建议如果有序集的第一个符号是 SKP，多通道链路的接收方在通道错误状态寄存器中为通道报告错误，如果接收到的符号编号 1 到 4N 与表 4-22 或表 4-23 中的对应符号不匹配
- 当前一个块包含 EDS 令牌时在任何通道上接收数据块是成帧错误。强烈建议多通道链路的接收方在通道错误状态寄存器中为收到数据块的通道报告错误
- 接收方允许检查不同通道上的不同有序集。例如，通道 0 收到 SKP 有序集，通道 1 收到 EIOS。如果检查，接收不同的有序集是成帧错误

###### 4.2.2.3.4 成帧错误恢复

如果接收方在处理数据流时检测到成帧错误，它必须：

- 按第 4.2.4.8 节所述报告接收方错误
- 停止处理数据流。当收到下一个 SDS 有序集时，按前述方式启动新数据流的处理
- 启动第 4.2.4.8 节所述的错误恢复过程。如果 LTSSM 状态为 L0，指示 LTSSM 进入 Recovery。如果检测到成帧错误时 LTSSM 状态为 Configuration.Complete 或 Configuration.Idle，错误恢复过程通过从 Configuration.Idle 到 Recovery.RcvrLock 的指定超时转换或通过 L0 到 Recovery 的转换来满足。如果检测到成帧错误时 LTSSM 状态为 Recovery.RcvrCfg 或 Recovery.Idle，错误恢复过程通过从 Recovery.Idle 到 Recovery.RcvrLock 的指定超时转换或通过从 L0 到 Recovery 的指示转换来满足。如果 LTSSM 子状态为 Recovery.RcvrLock 或 Configuration.Linkwidth.Start，错误恢复过程在退出这些子状态时满足，不需要指示 LTSSM 进入 Recovery
  - 注意：成帧错误恢复机制不期望直接导致任何数据链路层发起的恢复操作，例如 NAK

**实现说明：由于检测到成帧错误而在 Recovery 中花费的时间**

使用 128b/130b 编码时，所有成帧错误都需要链路恢复。预期实现从两个端口都进入 Recovery 状态的时间测量起，需要少于 1 微秒从成帧错误中恢复。

###### 4.2.2.4 加扰

多通道链路中发送方的每个通道可以实现单独的 LFSR 进行加扰。多通道链路中接收方的每个通道可以实现单独的 LFSR 进行解扰。实现可以选择实现更少的 LFSR，但必须实现与独立 LFSR 相同的功能。

LFSR 使用以下多项式：
```
G(X) = X^23 + X^21 + X^16 + X^8 + X^5 + X^2 + 1
```
如图 4-18 所示。加扰规则如下：

- 同步头的两位不加扰，不推进 LFSR
- 电气空闲退出有序集 (EIEOS) 的所有 16 个符号绕过加扰。加扰 LFSR 在 EIEOS 的最后一个符号传输后初始化，解扰 LFSR 在 EIEOS 的最后一个符号接收后初始化
- TS1 和 TS2 有序集：
  - TS1 或 TS2 有序集的符号 0 绕过加扰
  - 符号 1-13 加扰
  - 符号 14 和 15 如果需要直流平衡则绕过加扰，但如果不需要直流平衡则加扰
- 快速训练序列 (FTS) 有序集的所有 16 个符号绕过加扰
- 数据流起始 (SDS) 有序集的所有 16 个符号绕过加扰
- 电气空闲有序集 (EIOS) 的所有 16 个符号绕过加扰
- SKP 有序集的所有符号绕过加扰
- 发送方为所有有序集（SKP 有序集除外）的所有符号推进其 LFSR。SKP 有序集的任何符号不推进 LFSR
- 接收方评估有序集块的符号 0 以确定是否推进其 LFSR。如果块的符号 0 是 SKP（见第 4.2.7.2 节），则块的任何符号不推进 LFSR。否则，块的所有符号推进 LFSR
- 数据块的所有 16 个符号加扰并推进加扰器
- 对于需要加扰的符号，最低有效位首先加扰，最高有效位最后加扰
- LFSR 的种子值取决于链路首次进入 Configuration.Idle 时分配给通道的通道号（即从 Detect 经 Polling 进入，LinkUp = 0b）
  - 通道号模 8 的种子值为：
    - 通道 0：1DBFBCh
    - 通道 1：0607BBh
    - 通道 2：1EC760h
    - 通道 3：18C0DBh
    - 通道 4：010F12h
    - 通道 5：19CFC9h
    - 通道 6：0277CEh
    - 通道 7：1BB807h
- 只要 LinkUp=1，LFSR 的种子值就不会改变。通过 LTSSM Configuration 状态的链路重新配置不会修改初始通道号分配，只要 LinkUp 保持为 1（即使通道分配在 Configuration 期间可能改变）
- 使用 128b/130b 编码时，在 Configuration.Complete 中无法禁用加扰
- 环回从设备不得对环回位流进行解扰或加扰

## 第一章 引言

本章概述了 PCI Express 架构和关键概念。PCI Express 是一种高性能、通用的 I/O 互连标准，专为未来各种计算和通信平台而设计。PCI 的关键特性，如其使用模型、加载-存储架构和软件接口得以保留，而其并行总线实现则被高度可扩展的全串行接口所取代。PCI Express 利用点对点互连、交换技术和分组化协议的最新进展，提供全新的性能水平和功能特性。电源管理、服务质量 (QoS)、热插拔/热交换支持、数据完整性和错误处理是 PCI Express 支持的一些高级功能。

- • 对称性 - 每个 Link 必须在每个方向支持对称数量的 Lane，即 x16 Link 表示每个方向有 16 个差分信号对。

###### 1.3.1 根复用器 (Root Complex)

- • RC 表示连接 CPU/内存子系统与 I/O 的 I/O 层次结构的根。
- • 如图 1-2 所示，RC 可以支持一个或多个 PCI Express 端口。每个接口定义一个独立的层次结构域。每个层次结构域可以由单个端点或包含一个或多个交换机组件和端点的子层次结构组成。

- • 通过 RC 在层次结构域之间路由对等事务的能力是可选的，取决于实现。例如，实现可以在根复用器内部包含真实或虚拟交换机，以软件透明的方式启用完全的对等支持。与交换机的规则不同，RC 通常被允许在对等路由层次结构域之间的事务时将数据包拆分为更小的数据包（以下注明的例外情况除外），例如，将具有 256 字节载荷的单个数据包拆分为两个各 128 字节载荷的数据包。生成的数据包受本规范中包含的正常数据包形成规则约束（例如，Max_Payload_Size、读完成边界 (RCB) 等）。组件设计人员应注意，将数据包拆分为更小的数据包可能会产生负面的性能影响，特别是对于寻址 PCI Express 到 PCI/PCI-X 桥后设备的事务。

例外：支持 Vendor_Defined 消息对等路由的 RC 不允许将 Vendor_Defined 消息数据包拆分为更小的数据包，除非在 128 字节边界处（即，所有生成的数据包除最后一个外必须是 128 字节的整数倍长度），以保留跨 PCI Express 到 PCI/PCI-X 桥转发消息的能力。

- • RC 必须支持作为请求者生成配置请求。
- • RC 被允许支持作为请求者生成 I/O 请求。RC 被允许向选定的根端口的 80h 和 84h 位置生成 I/O 请求，而不考虑该根端口的 PCI 桥 I/O 解码配置；建议仅在特别需要时启用此机制。
- • RC 不得作为完成者支持锁定语义。
- • RC 被允许支持作为请求者生成锁定请求。


###### 1.3.2 端点 (Endpoints)

端点是指可以作为 PCI Express 事务的请求者或完成者的 Function 类型，可以代表自身或代表独立的非 PCI Express 设备（PCI 设备或主机 CPU 除外），例如 PCI Express 附加图形控制器或 PCI Express-USB 主机控制器。端点分为传统端点、PCI Express 端点或根复用器集成端点 (RCiEP)。

###### 1.3.2.1 传统端点规则

- • 传统端点必须是具有 Type 00h 配置空间头的 Function。
- • 传统端点必须作为完成者支持配置请求。
- • 传统端点可以作为完成者支持 I/O 请求。

◦ 传统端点被允许接受 80h 和 84h 位置（或两者）的 I/O 请求，而不考虑该端点的 I/O 解码配置。

- • 传统端点可以生成 I/O 请求。
- • 如果设备的传统软件支持需求需要，传统端点可以作为完成者支持锁定内存语义。
- • 传统端点不得发出锁定请求。
- • 传统端点可以实现扩展配置空间能力，但此类能力可能被软件忽略。
- • 作为内存事务请求者运行的传统端点不需要能够生成 4 GB 或更大的地址。
- • 如果请求中断资源，传统端点需要支持 MSI 或 MSI-X 或两者。如果实现了 MSI，传统端点被允许支持 MSI 能力结构的 32 位或 64 位消息地址版本。
- • 传统端点被允许为请求内存资源的基址寄存器支持 32 位寻址。


- • 传统端点必须出现在根复用器发起的层次结构域之一内。


###### 1.3.2.2 PCI Express 端点规则

- • PCI Express 端点必须是具有 Type 00h 配置空间头的 Function。
- • PCI Express 端点必须作为完成者支持配置请求。
- • PCI Express 端点不得依赖操作系统分配通过 BAR 声明的 I/O 资源。
- • PCI Express 端点不得生成 I/O 请求。
- • PCI Express 端点不得作为完成者支持锁定请求，也不得作为请求者生成锁定请求。PCI Express 兼容的软件驱动程序和应用程序必须编写为在访问 PCI Express 端点时防止使用锁定语义。
- • 作为内存事务请求者运行的 PCI Express 端点需要能够生成大于 4 GB 的地址。
- • 如果请求中断资源，PCI Express 端点需要支持 MSI 或 MSI-X 或两者。如果实现了 MSI，PCI Express 端点必须支持 MSI 能力结构的 64 位消息地址版本。
- • 通过 BAR 请求内存资源的 PCI Express 端点必须设置 BAR 的可预取位，除非该范围包含具有读取副作用的地址或 Function 不容忍写入合并的地址。有关设置可预取位的更多指导，请参见第 7.5.1.2.1 节。

- • 对于 PCI Express 端点，必须为所有设置了可预取位的 BAR 支持 64 位寻址。对于所有未设置可预取位的 BAR，允许 32 位寻址。

- • BAR 请求的最小内存地址范围为 128 字节。

- • PCI Express 端点必须出现在根复用器发起的层次结构域之一内。


###### 1.3.2.3 根复用器集成端点规则

- • 根复用器集成端点 (RCiEP) 实现在包含根端口的根复用器内部逻辑上。

- • RCiEP 必须是具有 Type 00h 配置空间头的 Function。

- • RCiEP 必须作为完成者支持配置请求。

- • RCiEP 不得要求通过 BAR 声明的 I/O 资源。

- • RCiEP 不得生成 I/O 请求。

- • RCiEP 不得作为完成者支持锁定请求，也不得作为请求者生成锁定请求。PCI Express 兼容的软件驱动程序和应用程序必须编写为在访问 RCiEP 时防止使用锁定语义。

- • 作为内存事务请求者运行的 RCiEP 需要能够生成等于或大于主机作为完成者能够处理的地址。

- • 如果请求中断资源，RCiEP 需要支持 MSI 或 MSI-X 或两者。如果实现了 MSI，RCiEP 被允许支持 MSI 能力结构的 32 位或 64 位消息地址版本。

- • RCiEP 被允许为请求内存资源的基址寄存器支持 32 位寻址。


- • RCiEP 不得在 PCI Express 扩展能力中实现 Link 能力、Link 状态、Link 控制、Link 能力 2、Link 状态 2 和 Link 控制 2 寄存器。

- • 如果 RCiEP 与可选的根复用器事件收集器相关联，它必须通过根复用器事件收集器发信号通知 PME 和错误条件。

- • RCiEP 不得与多个根复用器事件收集器相关联。

- • RCiEP 不实现活动状态电源管理。

- • RCiEP 不能独立于整个根复用器进行热插拔。

- • RCiEP 不得出现在根复用器公开的任何层次结构域中。

- • RCiEP 不得出现在交换机中。

###### 1.5.1 事务层

架构的上层是事务层。事务层的主要职责是组装和拆卸 TLP。TLP 用于传递事务，如读取和写入，以及某些类型的事件。事务层还负责管理 TLP 的基于信用的流控。

每个需要响应数据包的请求数据包都作为分离事务实现。每个数据包都有一个唯一标识符，使响应数据包能够定向到正确的发起者。数据包格式根据事务类型（内存、I/O、配置和消息）支持不同形式的寻址。数据包还可以具有诸如无嗅探、宽松排序和基于 ID 的排序 (IDO) 等属性。

事务层支持四个地址空间：它包括三个 PCI 地址空间（内存、I/O 和配置），并添加了消息空间。本规范使用消息空间支持所有先前的边带信号，如中断、电源管理请求等，作为带内消息事务。您可以将 PCI Express 消息事务视为"虚拟线路"，因为它们的效果是消除平台实现中当前使用的各种边带信号。

###### 1.5.2 数据链路层

堆栈中的中间层是数据链路层，作为事务层和物理层之间的中间阶段。数据链路层的主要职责包括 Link 管理和数据完整性，包括错误检测和错误纠正。

数据链路层的发送端接受事务层组装的 TLP，计算并应用数据保护码和 TLP 序列号，并将它们提交给物理层以便跨 Link 传输。接收数据链路层负责检查接收到的 TLP 的完整性，并将其提交给事务层进行进一步处理。在检测到 TLP 错误时，该层负责请求重传 TLP，直到信息正确接收，或者确定 Link 已失败。

数据链路层还生成和使用用于 Link 管理功能的数据包。为了将这些数据包与事务层使用的数据包 (TLP) 区分开来，当引用在数据链路层生成和使用的数据包时，将使用术语数据链路层数据包 (DLLP)。

###### 1.5.3 物理层

物理层包括接口操作的所有电路，包括驱动器和输入缓冲器、并转串和串转并转换器、PLL 和阻抗匹配电路。它还包括与接口初始化和维护相关的逻辑功能。物理层以特定于实现的格式与数据链路层交换信息。该层负责将从数据链路层接收的信息转换为适当的序列化格式，并以与连接到 Link 另一侧的设备兼容的频率和宽度跨 PCI Express Link 传输。

PCI Express 架构具有"钩子"，可通过速度升级和高级编码技术支持未来的性能增强。未来的速度、编码技术或介质可能仅影响物理层定义。

###### 1.5.4 层功能和服务

###### 1.5.4.1 事务层服务

事务层在生成和接收 TLP 的过程中，与 Link 另一侧的对应事务层交换流控信息。它还负责支持软件和硬件发起的电源管理。

初始化和配置功能要求事务层：

- • 存储由处理器或管理设备生成的 Link 配置信息
- • 存储由物理层硬件协商宽度和运行频率生成的 Link 能力

事务层的数据包生成和处理服务要求它：

- • 从设备核心请求生成 TLP
- • 将接收到的请求 TLP 转换为设备核心的请求
- • 将接收到的完成数据包转换为可传递给核心的载荷或状态信息
- • 检测不支持的 TLP 并调用适当的处理机制
- • 如果支持端到端数据完整性，则生成端到端数据完整性 CRC 并相应更新 TLP 头部。


流控服务：

- • 事务层跟踪跨 Link 的 TLP 流控信用。
- • 使用数据链路层的传输服务定期将事务信用状态传输到远程事务层。
- • 远程流控信息用于限制 TLP 传输。


排序规则：

- • 符合 PCI/PCI-X 的生产者/消费者排序模型
- • 支持宽松排序的扩展

- • 支持基于 ID 的排序扩展

电源管理服务：

- • 通过机制进行软件控制的电源管理，由系统软件决定。
- • 硬件控制的自主电源管理在全开电源状态下最小化功耗。

虚拟通道和流量类型：

- • 虚拟通道机制和流量类型标识的组合用于支持差异化服务和某些应用类别的 QoS 支持。
- • 虚拟通道：虚拟通道提供一种在 Link 的给定公共物理资源上支持多个独立逻辑数据流的手段。概念上，这涉及将不同的数据流复用到单个物理 Link 上。


- • 流量类型：流量类型是事务层数据包标签，通过 Fabric 端到端传输时不被修改。在 Fabric 内的每个服务点（例如，交换机），流量类型标签用于应用适当的服务策略。每个流量类型标签定义一个唯一的排序域 - 对于包含不同流量类型标签的数据包，不提供排序保证。


###### 1.5.4.2 数据链路层服务

数据链路层负责与其 Link 对端的对应方可靠地交换信息。

初始化和电源管理服务：

- • 接受来自事务层的电源状态请求并传递给物理层
- • 将活动/复位/断开/电源管理状态传递给事务层

数据保护、错误检查和重试服务：

- • CRC 生成
- • 存储已发送的 TLP 以用于数据链路层重试
- • 错误检查
- • TLP 确认和重试消息
- • 用于错误报告和记录的错误指示


###### 1.5.4.3 物理层服务

接口初始化、维护控制和状态跟踪：

- • 复位/热插拔控制/状态
- • 互连电源管理
- • 宽度和 Lane 映射协商
- • Lane 极性反转

符号和特殊有序集生成：

- • 8b/10b 编码/解码
- • 嵌入时钟调谐和对齐

符号传输和对齐：

- • 发送电路
- • 接收电路

###### 2.2 事务层协议 - 数据包定义

PCI Express 使用基于数据包的协议在通过链路相互通信的两个组件的事务层之间交换信息。PCI Express 支持以下基本事务类型：内存、I/O、配置和消息。支持两种内存请求寻址格式：32 位和 64 位。

事务通过请求和完成来承载。完成仅在需要时使用，例如，返回读取数据，或确认 I/O 和配置写事务的完成。通过数据包头中 Transaction ID 字段的值将完成与其对应的请求相关联。

所有标记为 Reserved（有时缩写为 R）的 TLP 字段在形成 TLP 时必须填充全 0。接收方必须忽略此类字段中的值，交换机必须不加修改地转发这些值。请注意，某些字段既有规定值也有 Reserved 值 - 这些情况下 Reserved 值的处理在每种情况下单独规定。

###### 2.2.2 带数据载荷的 TLP - 规则

- • 长度以整数 DW 为单位指定
- • Length[9:0] 对于所有消息均为 Reserved，除非消息明确引用数据长度

◦ 参见第 2.2.8 节中的消息代码表。

- • 带数据载荷的 TLP 发送方不得允许 TLP 的 Length 字段给出的数据载荷长度超过发送方 Device Control 寄存器中 Max_Payload_Size 字段指定的值（以整数 DW 为单位）（见第 7.5.3.4 节）。

- ◦ 对于 ARI 设备，Max_Payload_Size 仅由 Function 0 中的设置决定。其他 Function 中的 Max_Payload_Size 设置被忽略。

- ◦ 对于与非 ARI 多功能设备 (MFD) 关联的上游端口，如果所有 Function 的 Max_Payload_Size 设置相同，则发送的 TLP 数据载荷不得超过公共 Max_Payload_Size 设置。

- ◦ 对于与非 ARI MFD 关联的上游端口，如果所有 Function 的 Max_Payload_Size 设置不相同，则发送的 TLP 数据载荷不得超过实现特定的 Max_Payload_Size 设置。

- ▪ 鼓励发送方实现使用产生事务的 Function 的 Max_Payload_Size 设置，或所有 Function 中最小的 Max_Payload_Size 设置。

- ▪ 软件不应将不同 Function 中的 Max_Payload_Size 设置为不同值，除非软件了解特定实现。


- ◦ 注意：Max_Payload_Size 仅适用于带数据载荷的 TLP；内存读请求的长度不受 Max_Payload_Size 限制。内存读请求的大小由 Length 字段控制。


- • 接收的 TLP 的数据载荷大小（由 TLP 的 Length 字段给出）不得超过接收方 Device Control 寄存器中 Max_Payload_Size 字段指定的值（以整数 DW 为单位）（见第 7.5.3.4 节）。


- ◦ 接收方必须检查是否违反此规则。如果接收方确定 TLP 违反此规则，则该 TLP 为畸形 TLP。

▪ 这是与接收端口相关的报告错误（见第 6.2 节）。

- ◦ 对于 ARI 设备，Max_Payload_Size 仅由 Function 0 中的设置决定。其他 Function 中的 Max_Payload_Size 设置被忽略。

- ◦ 对于与非 ARI MFD 关联的上游端口，如果所有 Function 的 Max_Payload_Size 设置相同，则接收方需要根据公共 Max_Payload_Size 设置检查 TLP 数据载荷大小。


- ◦ 对于与非 ARI MFD 关联的上游端口，如果所有 Function 的 Max_Payload_Size 设置不相同，则接收方需要根据实现特定的 Max_Payload_Size 设置检查 TLP 数据载荷。


- ▪ 鼓励接收方实现使用事务目标 Function 的 Max_Payload_Size 设置，或所有 Function 中最大的 Max_Payload_Size 设置。

- ▪ 软件不应将不同 Function 中的 Max_Payload_Size 设置为不同值，除非软件了解特定实现。


- • 对于包含数据的 TLP，Length 字段中的值与 TLP 中实际包含的数据量必须匹配。

◦ 接收方必须检查是否违反此规则。如果接收方确定 TLP 违反此规则，则该 TLP 为畸形 TLP。

▪ 这是与接收端口相关的报告错误（见第 6.2 节）。

- • Length 字段中的值仅适用于数据 - TLP 摘要不包含在长度中
- • 当与字节地址关联的数据载荷包含在 TLP（AtomicOp 请求或 AtomicOp 完成除外）中时，头之后的数据的第一个字节对应于最接近零的字节地址，后续字节按字节地址递增顺序排列。

◦ 示例：对于向位置 100h 的 16 字节写入，头之后的第一个字节是要写入位置 100h 的字节，第二个字节将写入位置 101h，依此类推，最后一个字节写入位置 10Fh。

- • AtomicOp 请求和 AtomicOp 完成中的数据载荷必须格式化，使得 TLP 头之后的数据第一个字节是第一个数据值的最低有效字节，后续数据字节按显著性严格递增顺序排列。对于 Compare And Swap (CAS) 请求，第二个数据值紧跟第一个数据值，必须采用相同格式。


- ◦ AtomicOp Completer 用于在目标位置读取和写入数据的字节序格式是实现特定的，允许 Completer 确定适合目标内存的任何格式（例如，小端、大端等。）AtomicOp Completer 的字节序格式能力报告和控制不在本规范范围内。
- ◦ 小端示例：对于目标内存为小端格式的 64 位（8 字节）Swap 请求，目标位置为 100h，头之后的第一个字节写入位置 100h，第二个字节写入位置 101h，依此类推，最后一个字节写入位置 107h。请注意，在执行写入之前，Completer 首先读取目标内存位置，以便可以在完成中返回原始值。完成中的字节地址与数据的对应关系与请求中相同。
