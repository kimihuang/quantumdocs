PCI Express Base 5.0 规范文档结构概览
==========================================

本文档基于 PCI Express Base Specification Revision 5.0 Version 1.0 深入分析，
参考文档：docs/pdf/PCIE/PCI_Express_Base_5.0r1.0.md。

.. contents::
   :depth: 2
   :local:


引言
----

PCI Express（简称 PCIe）是一种高性能、低延迟的串行互连技术，用于连接计算机系统中的处理器、内存和各种外设。作为第三代 I/O 互连标准，PCIe 取代了传统的并行 PCI 总线，采用点对点串行架构，提供了更高的带宽和更好的可扩展性。

第三代 I/O 互连技术
~~~~~~~~~~~~~~~~~~~~

传统的并行总线（如 PCI 和 PCI-X）在高频率下面临严重的信号完整性问题：时钟偏移、串扰、信号衰减等限制了带宽提升。PCIe 采用串行差分信号传输，通过嵌入式时钟和点对点拓扑解决了这些问题。

PCIe 的核心设计理念包括：

**点对点架构**
每个设备独占一条链路，不存在总线竞争。这与传统共享总线不同，传统架构中所有设备共享带宽，当多个设备同时通信时会产生仲裁延迟。点对点架构确保每条链路的带宽完全由连接的两个设备独享。

**差分信号传输**
使用两根信号线（D+ 和 D-）传输互补信号，接收端通过检测两线之间的电压差来恢复数据。这种方式的抗干扰能力远强于单端信号，因为外部噪声会同时影响两根线，在差分计算中被抵消。差分信号还支持更低的电压摆幅，减少功耗和 EMI。

**嵌入式时钟**
数据与时钟一起编码传输，接收端从数据流中恢复时钟。这消除了传统并行总线中时钟与数据之间的偏移问题，也称为时钟数据恢复（CDR）技术。8b/10b 和128b/130b 编码都包含足够的跳变边沿供 CDR 电路锁定。

PCI Express 链路
~~~~~~~~~~~~~~~~~

PCIe 链路是两个组件之间的物理连接，由一个或多个通道（Lane）组成。每个通道包含两对差分信号线：

- **发送方向（TX）**：一对差分线，用于向对方发送数据
- **接收方向（RX）**：一对差分线，用于从对方接收数据

这种双单工架构允许同时进行双向通信，每个方向独立传输。与传统半双工总线相比，PCIe 的有效带宽翻倍。

**通道聚合**

PCIe 支持多通道绑定以扩展带宽。常见的链路宽度配置包括：

- x1：单通道，最小配置
- x2、x4：中等带宽需求
- x8、x16：高性能应用（如显卡）
- x32：服务器和高性能计算

多个通道并行传输数据字节，例如 x16 链路每个时钟周期传输 16 字节（128位）。接收端需要去歪斜逻辑来对齐各通道的数据，因为不同通道的传播延迟略有差异。

**数据传输速率演进**

PCIe 规范定义了多代数据传输速率：

.. list-table::
   :header-rows: 1
   :widths: 20 25 55

   * - 代次
     - 速率
     - 说明

   * - Gen 1
     - 2.5 GT/s
     - 原始带宽每通道约 250 MB/s（8b/10b 编码开销 20%）

   * - Gen 2
     - 5.0 GT/s
     - 带宽翻倍，每通道约 500 MB/s

   * - Gen 3
     - 8.0 GT/s
     - 采用 128b/130b 编码，编码效率提升至 98.5%

   * - Gen 4
     - 16.0 GT/s
     - 带宽再翻倍，每通道约 2 GB/s

   * - Gen 5
     - 32.0 GT/s
     - 本规范最高速率，每通道约 4 GB/s

PCI Express 拓扑结构
~~~~~~~~~~~~~~~~~~~~~

PCIe 采用层次化的树形拓扑结构，以根复合体为根节点，端点为叶节点，交换机作为分支节点。这种结构源自传统 PCI 总线的软件兼容性需求。

根复合体（Root Complex）
^^^^^^^^^^^^^^^^^^^^^^^^^

根复合体是 PCIe 层次结构的根节点，通常集成在 CPU 或芯片组中。其主要职责包括：

**系统接口**
连接 CPU 和主内存子系统，处理来自 CPU 的 I/O 请求，将内存访问转换为 PCIe 事务。

**层次结构管理**
初始化和管理整个 PCIe 设备树，负责设备枚举、资源分配和热插拔事件处理。

**配置空间访问代理**
响应来自 CPU 的配置访问请求，生成配置 TLP 发送到目标设备。

**地址路由**
根据内存映射和总线号范围，将请求路由到正确的下游端口或本地资源。

根复合体可以包含一个或多个根端口，每个根端口连接一条独立的 PCIe 链路。

端点（Endpoint）
^^^^^^^^^^^^^^^^^

端点是 PCIe 层次结构中的叶节点，代表实际的 I/O 设备。端点分为三类：

**传统端点（Legacy Endpoint）**

这类端点需要与传统 PCI 软件兼容，支持：

- INTx 中断信号模拟（通过消息传递）
- 传统 I/O 空间访问
- 可能支持 64 位 BAR 但非必须
- 最大 4KB 的有效载荷大小（MPS）

传统端点适用于需要与旧版操作系统或驱动程序兼容的场景。

**PCI Express 端点**

完全遵循 PCIe 规范的端点，具有以下特性：

- 必须支持 MSI 或 MSI-X 中断
- 所有 BAR 必须支持 64 位地址（对于内存 BAR）
- 支持最大 4KB MPS
- 必须支持设备序列号扩展能力

这类端点能充分利用 PCIe 的高级特性，性能更优。

**根复合体集成端点（RCiEP）**

直接集成在根复合体内部的端点，特点是：

- 不通过根端口连接
- 不参与链路训练（没有物理链路）
- 配置空间直接映射到系统地址空间
- 常用于平台特定功能（如管理引擎、信任平台模块）

交换机（Switch）
^^^^^^^^^^^^^^^^

交换机允许多个下游设备共享一个上游连接，扩展 PCIe 层次结构。从软件角度看，交换机表现为多个虚拟 PCI 桥的集合。

**内部结构**

交换机包含：

- 一个上游端口：连接根复合体或其他交换机
- 多个下游端口：连接端点或其他交换机
- 内部路由逻辑：根据地址或 ID 路由 TLP

**路由机制**

交换机维护路由表，根据 TLP 类型选择路由方式：

- 内存/IO 请求：地址路由，查找 BAR 和解码范围
- 配置请求：ID 路由，根据总线号转发
- 完成包：ID 路由，返回给请求者

交换机还支持对等通信（Peer-to-Peer），允许下游端口之间直接通信，无需经过根复合体。

根复合体事件收集器（RCEC）
^^^^^^^^^^^^^^^^^^^^^^^^^^^

RCEC 是一种特殊组件，用于收集和管理 RCiEP 的错误和事件。由于 RCiEP 没有物理链路，无法使用标准的错误报告机制，RCEC 提供了统一的错误处理接口。

RCEC 通过关联位图（Association Bitmap）指定哪些 RCiEP 属于其管理范围，当这些端点产生错误或事件时，RCEC 统一上报给系统。

PCI Express 分层架构
~~~~~~~~~~~~~~~~~~~~~

PCIe 协议栈采用三层结构，每层负责特定功能，层间通过明确定义的接口交互。这种分层设计提供了良好的模块化和实现灵活性。

事务层（Transaction Layer）
^^^^^^^^^^^^^^^^^^^^^^^^^^^^

事务层是协议栈的最高层，负责处理所有事务层包（TLP）的生成、解析和管理。

**核心功能**

- **TLP 封装与解封**：将请求和完成信息封装成 TLP，或从接收的 TLP 中提取信息
- **流量控制**：管理各虚拟通道的信用，防止缓冲区溢出
- **事务排序**：根据排序规则处理 TLP 的顺序
- **虚拟通道管理**：支持多个独立的数据流

**TLP 类型**

事务层处理多种 TLP 类型：

- 内存读写请求/完成
- I/O 读写请求/完成
- 配置读写请求/完成
- 消息请求（中断、电源管理等）

数据链路层（Data Link Layer）
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

数据链路层位于事务层和物理层之间，提供 TLP 的可靠传输服务。

**核心功能**

- **序列号管理**：为每个 TLP 分配 12 位序列号
- **LCRC 生成/校验**：添加链路层 CRC 保护数据完整性
- **重传机制**：检测丢失或损坏的 TLP 并请求重传
- **链路管理**：处理链路状态转换和电源管理

**DLLP 类型**

数据链路层包（DLLP）不携带有效载荷，仅用于链路管理：

- **Ack/Nak**：确认或否定确认 TLP 传输
- **流控更新**：向对方报告可用缓冲区空间
- **电源管理**：请求或确认电源状态转换

物理层（Physical Layer）
^^^^^^^^^^^^^^^^^^^^^^^^^^

物理层负责数据的物理传输，分为逻辑子块和电气子块。

**逻辑子块**

- **编码/解码**：8b/10b（Gen1/2）或 128b/130b（Gen3+）
- **加扰/解扰**：减少 EMI，平均化频谱
- **链路训练**：初始化链路、协商参数
- **通道管理**：处理通道绑定和去歪斜

**电气子块**

- **发送器**：驱动差分信号，支持去加重和预加重
- **接收器**：检测信号，支持均衡器
- **阻抗匹配**：确保信号完整性


事务层规范
----------

事务层是 PCIe 协议栈的核心，处理所有数据传输请求和响应。理解事务层对于掌握 PCIe 工作原理至关重要。

事务层概述
~~~~~~~~~~~~

事务层的主要职责是将来自设备核心或上层协议的请求转换为 TLP 发送到链路上，同时解析接收的 TLP 并传递给相应的处理模块。

**请求-完成模型**

PCIe 采用分离事务模型，请求和响应可以异步进行：

1. 请求者发送请求包（如内存读请求）
2. 完成者接收请求，处理并返回完成包
3. 请求者接收完成包，获取所需数据

这种模型允许在等待响应期间链路传输其他事务，提高链路利用率。与传统同步总线相比，分离事务模型显著提升了系统吞吐量。

**流量类别（Traffic Class, TC）**

PCIe 定义了 8 个流量类别（TC0-TC7），用于区分不同优先级的流量：

- TC0 是默认值，保证无拥塞情况下的公平服务
- 高 TC 值通常用于低延迟或高带宽需求的应用
- TC 通过 TLP 头的 TC 字段标记

TC 与虚拟通道（VC）的映射由配置决定，多个 TC 可以映射到同一个 VC。

地址空间和事务类型
~~~~~~~~~~~~~~~~~~~

PCIe 定义了四种独立的地址空间，每种空间有特定的用途和访问方式。

内存事务
^^^^^^^^

内存事务是 PCIe 最常用的操作，用于访问设备上的内存映射区域。

**地址宽度**

- **32 位地址**：访问 4GB 以内的地址空间，使用 3DW 头
- **64 位地址**：访问完整 64 位地址空间，使用 4DW 头

64 位地址支持对于访问大容量显存的显卡或高容量 NVMe SSD 至关重要。

**读写语义**

- **Memory Read**：请求者请求数据，完成者返回数据
- **Memory Write**：请求者发送数据，完成者接收并确认

**原子操作**

PCIe 2.0 引入的原子操作允许在设备内存上执行不可中断的读-修改-写操作：

- **FetchAdd**：读取当前值，加上指定值，返回原值
- **Unconditional Swap**：写入新值，返回原值
- **Compare and Swap (CAS)**：如果当前值等于比较值则写入新值，返回原值

原子操作对于无锁数据结构和多设备同步非常有用，避免了传统两阶段锁协议的开销。

I/O 事务
^^^^^^^^

I/O 事务支持传统 x86 I/O 端口空间访问，仅支持 32 位地址。现代设备很少使用 I/O 空间，主要保留用于兼容传统设备。

**限制**

- 最大传输长度为 4 字节（一个 DWORD）
- 不支持 64 位地址
- 不支持原子操作

配置事务
^^^^^^^^

配置事务用于访问设备的配置空间，每个功能（Function）有 4KB 配置空间。

**配置类型**

- **Type 0**：目标设备在当前总线，不转发
- **Type 1**：目标设备在其他总线，需要路由

配置请求使用 ID 路由，目标由总线号、设备号、功能号指定。

**配置空间布局**

.. code-block:: text

   Offset 0x000-0x0FF: PCI 兼容配置空间（前 64 字节为配置头）
   Offset 0x100-0xFFF: PCIe 扩展配置空间

消息事务
^^^^^^^^

消息事务是 PCIe 的创新特性，将传统 PCI 的边带信号（如中断）转换为带内消息传递。

**消息类型**

- **INTx 中断模拟**：Assert_INTx / Deassert_INTx
- **电源管理**：PM_PME、PME_To_Ack
- **错误报告**：ERR_COR、ERR_NONFATAL、ERR_FATAL
- **厂商定义消息**：Vendor_Defined Messages

消息机制的优点是减少引脚数量，简化板级设计，同时支持更灵活的拓扑结构。

数据包格式概述
~~~~~~~~~~~~~~~

TLP 是事务层传输的基本单元，由多个字段组成，每个字段有特定含义。

**TLP 结构**

.. code-block:: text

   +------------------+
   |   TLP Prefix     | (可选，用于扩展功能)
   +------------------+
   |   TLP Header     | (3DW 或 4DW，包含路由和控制信息)
   +------------------+
   |   Data Payload   | (可选，最大 4KB)
   +------------------+
   |   ECRC           | (可选，端到端 CRC)
   +------------------+

**TLP 头格式**

TLP 头长度取决于地址类型：

- 3DW 头：32 位地址请求、ID 路由请求、完成包
- 4DW 头：64 位地址请求

公共包头字段包括：

- **Fmt[2:0]**：格式字段，指示头长度和是否有数据
- **Type[4:0]**：事务类型
- **TC[2:0]**：流量类别
- **Attr[1:0]**：属性字段（宽松排序、无探测）
- **TD**：TLP Digest 存在标志
- **EP**：错误转发（数据污染）标志
- **Length[9:0]**：数据长度，以 DWORD 为单位
- **Requester ID**：请求者标识（Bus/Device/Function）
- **Tag**：请求标识，用于匹配完成包

路由和寻址规则
~~~~~~~~~~~~~~~

TLP 路由是 PCIe 架构的核心机制，决定了数据包如何从源到达目的地。

地址路由
^^^^^^^^

地址路由用于内存和 I/O 事务，基于目标地址转发。

**64 位地址路由**

.. code-block:: text

   Header DW 2: Address[63:32]
   Header DW 3: Address[31:2], 00

地址字段对齐到 4 字节边界，最低 2 位保留。

**路由过程**

每个桥接设备（包括根端口和交换机端口）维护地址解码范围：

1. 比较 TLP 地址与各端口的 BAR 和解码范围
2. 如果地址落入某端口范围，转发到该端口
3. 如果无匹配且是 Type 1 配置，可能转发到次要总线

ID 路由
^^^^^^^

ID 路由用于配置事务和完成包，基于 Bus/Device/Function 号转发。

**路由 ID 格式**

.. code-block:: text

   Bus Number[7:0]: Device Number[4:0]: Function Number[2:0]

**路由过程**

1. 比较目标总线号与端口的总线号范围
2. 如果目标总线等于端口的次要总线号，在该总线上进行 Type 0 配置
3. 如果目标总线在端口的从属总线范围内，作为 Type 1 转发
4. 否则向上游转发

流量控制机制
~~~~~~~~~~~~

流量控制防止发送方淹没接收方的缓冲区，是 PCIe 可靠传输的基础。

**信用机制**

每个虚拟通道独立维护信用：

- **PH（Posted Header）**：发布请求头的信用
- **PD（Posted Data）**：发布请求数据的信用
- **NPH（Non-Posted Header）**：非发布请求头的信用
- **NPD（Non-Posted Data）**：非发布请求数据的信用
- **CplH（Completion Header）**：完成包头的信用
- **CplD（Completion Data）**：完成包数据的信用

**信用单位**

信用以 16 字节（一个数据块）为单位计算。例如，发送一个 128 字节的内存读请求消耗 8 个 CplD 信用。

**流控过程**

1. 接收方在链路初始化时通告初始信用
2. 发送方跟踪可用信用，仅在有足够信用时发送 TLP
3. 接收方处理完 TLP 后，通过 Update_FC DLLP 返回信用
4. 发送方收到信用更新后增加可用信用计数

事务排序规则
~~~~~~~~~~~~

PCIe 定义了严格的排序规则，确保事务以可预测的顺序执行，同时允许优化以提高性能。

**基本排序规则**

.. list-table::
   :header-rows: 1
   :widths: 20 20 20 20 20

   * - 前一行
     - Posted 请求
     - Non-Posted 请求
     - 完成包
     - 说明

   * - Posted 请求
     - 否
     - 否
     - 是
     - 写请求可以彼此乱序

   * - Non-Posted 请求
     - 是
     - 否
     - 是
     - 读请求必须在写之后

   * - 完成包
     - 是
     - 否
     - 是
     - 响应必须在请求之后

**宽松排序属性**

设置 RO（Relaxed Ordering）属性的 TLP 可以绕过某些排序约束，允许更灵活的调度，提高性能。例如，RO 允许读完成包跨越之前的写请求。

**ID 基排序属性**

IDO（ID-Based Ordering）属性启用后，来自不同请求者的事务可以乱序，但同一请求者的事务保持顺序。这对于多核系统特别有用，允许不同核心的请求并行处理。


数据链路层规范
--------------

数据链路层提供 TLP 的可靠传输，是 PCIe 高可靠性的关键保障。

数据链路层概述
~~~~~~~~~~~~~~

数据链路层位于事务层和物理层之间，对上层提供透明可靠的数据传输服务。

**核心职责**

- **完整性保护**：为 TLP 添加 LCRC 和序列号
- **错误检测与恢复**：检测传输错误并触发重传
- **链路状态管理**：维护链路的操作状态
- **流控初始化**：初始化流控信用交换

**服务接口**

数据链路层向上层提供的服务：

- **TLP 传输**：可靠发送和接收 TLP
- **链路状态通知**：通知上层链路状态变化
- **电源管理支持**：参与链路电源状态转换

数据链路控制和管理状态机
~~~~~~~~~~~~~~~~~~~~~~~~~~

DLCMSM（Data Link Control and Management State Machine）管理链路的生命周期。

**状态定义**

- **DL_Inactive**：链路未激活或物理链路未建立

  - TLP 和 DLLP 都不能传输
  - 等待物理层链路训练完成

- **DL_Init**：链路初始化阶段

  - 初始化流控信用
  - 交换数据链路特征
  - 分为 DL_Init_FCU（流控初始化）和 DL_Init_FCDLC（特征交换）子状态

- **DL_Active**：链路正常工作

  - TLP 和 DLLP 正常传输
  - 重传机制激活

**状态转换**

.. code-block:: text

   Reset → DL_Inactive → DL_Init → DL_Active
                            ↓
                        DL_Inactive (错误恢复)

数据链路特征交换
~~~~~~~~~~~~~~~~

数据链路特征交换用于协商双方支持的可选特性。

**协商的特征**

- **扩展标签支持**：使用 8 位而非 5 位标签
- **10 位标签支持**：Gen3+ 的更大标签空间
- **重放定时器缩放**：调整重传超时参数
- **数据速率**：协商共同支持的最高速率

特征交换通过数据链路特征（DLF）DLLP 完成，双方交换能力位图并取交集。

流控初始化协议
~~~~~~~~~~~~~~

流控初始化建立初始信用值，是链路正常工作的前提。

**初始化流程**

1. 进入 DL_Init_FCU 状态
2. 接收方准备初始信用值并存储
3. 发送 InitFC1 DLLP 通告信用
4. 发送 InitFC2 DLLP 确认
5. 重复直到所有 VC 初始化完成

**信用初始化规则**

- 初始信用反映接收缓冲区的实际容量
- 发送方在收到初始信用前不能发送任何 TLP
- 信用值以 16 字节为单位编码

数据链路层包（DLLP）
~~~~~~~~~~~~~~~~~~~~

DLLP 是数据链路层的控制包，不携带用户数据，仅用于链路管理。

**DLLP 类型**

- **Ack DLLP**：确认成功接收的 TLP 序列号

  - 格式：Ack[11:0] 序列号
  - 表示该序列号及之前的所有 TLP 已正确接收

- **Nak DLLP**：请求重传

  - 格式：Nak[11:0] 序列号
  - 指示从该序列号开始重传

- **UpdateFC DLLP**：流控信用更新

  - 格式：各类型信用增量
  - 允许发送方发送更多数据

- **PM DLLP**：电源管理

  - Enter_L1、Enter_L23 等状态转换请求

- **Vendor Specific DLLP**：厂商自定义

**DLLP 格式**

.. code-block:: text

   +--------+--------+--------+--------+
   |  Type  |   Data/Content         |
   +--------+--------+--------+--------+
   |         16-bit CRC               |
   +--------+--------+--------+--------+

DLLP 固定 6 字节，包括 1 字节类型、4 字节内容和 2 字节 CRC。

数据完整性机制
~~~~~~~~~~~~~~

数据链路层实现了完整的数据完整性保护机制。

LCRC 和序列号管理
~~~~~~~~~~~~~~~~~~

**序列号**

12 位序列号范围 0-4095，循环使用：

- 发送方为每个 TLP 分配递增序列号
- 接收方通过序列号检测丢失和重复
- 序列号位于 TLP 头之后、数据之前

**LCRC 计算**

32 位 LCRC 使用标准多项式，覆盖：

- TLP 头
- 数据（如果有）
- 序列号

LCRC 由发送方计算并附加到 TLP 末尾，接收方重新计算并验证。

重传机制
~~~~~~~~

重传机制确保 TLP 的可靠传输，处理传输错误。

**发送方行为**

1. 保存每个发送的 TLP 到重传缓冲区
2. 启动重传定时器
3. 收到 Ack 后释放确认的 TLP
4. 收到 Nak 或超时后重传

**接收方行为**

1. 检查每个接收 TLP 的 LCRC
2. 检查序列号是否连续
3. 正确接收则发送 Ack
4. 错误或丢失则发送 Nak

**重传定时器**

- **replay_timer**：等待 Ack/Nak 的超时值
- **replay_timer_limit**：最大重传次数
- 超过限制触发链路重新训练


物理层逻辑块
------------

物理层负责数据的物理传输，包括编码、链路训练和电气接口。

物理层概述
~~~~~~~~~~

物理层分为逻辑子块和电气子块，各自承担不同职责。

**逻辑子块功能**

- 数据编码和解码
- 加扰和解扰
- 链路训练和状态管理
- 通道对齐和去歪斜

**电气子块功能**

- 发送器驱动电路
- 接收器检测和均衡
- 阻抗匹配
- ESD 保护

编码机制
~~~~~~~~

PCIe 使用不同的编码方案适配不同数据速率的需求。

8b/10b 编码（2.5 GT/s 和 5.0 GT/s）
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

8b/10b 编码将 8 位数据编码为 10 位码字，由 IBM 开发。

**编码原理**

- 将 8 位数据分为 5 位和 3 位两部分
- 分别使用 5b/6b 和 3b/4b 编码
- 总共 10 位输出

**特性**

- **直流平衡**：确保 0 和 1 的数量大致相等，便于交流耦合
- **跳变密度**：保证足够的跳变边沿供时钟恢复
- **特殊符号**：定义 K 码用于控制功能（如帧定界）

**开销**

8b/10b 编码有 20% 的开销，即实际有效带宽为原始带宽的 80%。

例如，5.0 GT/s 链路：
- 原始带宽：5.0 Gbps
- 编码后有效带宽：4.0 Gbps = 500 MB/s（每通道）

128b/130b 编码（8.0 GT/s 及更高）
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

128b/130b 编码用于 Gen3 及更高速度，显著降低编码开销。

**编码原理**

- 将 128 位数据块添加 2 位同步头
- 同步头标识块类型：数据块或有序集块
- 不执行实际的数据转换，仅添加开销位

**块格式**

数据块：
.. code-block:: text

   | Sync (2b) | Data Payload (128b) |
   | 10b or 01b| Lane 0-15 bytes     |

有序集块：
.. code-block:: text

   | Sync (2b) | Ordered Set Type | Payload |
   | 10b       | 8b               | 112b    |

**优势**

- 编码开销仅 1.54%（2/130），相比 8b/10b 的 20% 大幅降低
- 简化的编码/解码逻辑
- 更适合高速传输

**加扰**

128b/130b 必须使用加扰来减少 EMI：

- 使用线性反馈移位寄存器（LFSR）生成伪随机序列
- 数据与加扰序列异或运算
- 接收端使用相同种子解扰

加扰确保数据模式的随机性，即使传输全 0 或全 1 数据，信号也具有足够的跳变密度。

链路均衡过程
~~~~~~~~~~~~

链路均衡是 Gen3 及更高速度的关键技术，补偿通道损耗和码间干扰（ISI）。

**发送端均衡**

发送端可以调整信号波形来预补偿通道损耗：

- **预加重（Pre-cursor）**：增强跳变前的信号
- **去加重（De-emphasis）**：降低跳变后的信号幅度
- **主标度**：正常信号电平

均衡系数通过链路训练协商确定。

**接收端均衡**

接收端使用多种均衡技术恢复信号：

- **CTLE（连续时间线性均衡器）**：高频提升
- **DFE（判决反馈均衡器）**：消除码间干扰

**均衡阶段**

链路均衡分四个阶段：

1. **Phase 0**：使用默认设置建立通信
2. **Phase 1**：下游端口调整发送设置
3. **Phase 2**：上游端口调整发送设置
4. **Phase 3**：确认最终设置

链路初始化和训练
~~~~~~~~~~~~~~~~

链路训练是建立 PCIe 链路通信的过程，由硬件自动完成。

**训练目标**

- 建立位锁定和符号锁定
- 协商链路宽度和数据速率
- 实现通道去歪斜
- 配置均衡参数

**训练序列（Training Sequence）**

- **TS1/TS2 有序集**：携带链路配置信息

  - 链路号、通道号
  - 支持的数据速率
  - 均衡请求

- **电气空闲有序集（EIOS）**：进入低功耗状态

- **快速训练序列（FTS）**：L0s 状态快速恢复

链路训练状态机（LTSSM）
~~~~~~~~~~~~~~~~~~~~~~~~

LTSSM 是物理层的核心状态机，控制链路从初始化到正常工作的全过程。

Detect（检测）状态
^^^^^^^^^^^^^^^^^^

检测链路对端是否存在。

**Detect.Quiet**
- 等待接收器检测稳定
- 检测到接收端的存在

**Detect.Active**
- 驱动检测脉冲
- 检测对端接收器的阻抗
- 成功则进入 Polling

**接收器检测原理**

发送端通过检测接收端终端电阻（约 40-60Ω）判断对端是否存在。检测机制使用小幅度脉冲测量阻抗变化。

Polling（轮询）状态
^^^^^^^^^^^^^^^^^^^

建立链路的基本通信能力。

**Polling.Active**
- 发送 TS1 有序集
- 建立位锁定和符号锁定（8b/10b）或块锁定（128b/130b）
- 持续至少 24 ms

**Polling.Compliance**
- 可选的合规测试模式
- 发送特定测试模式
- 用于一致性测试

**Polling.Configuration**
- 交换 TS2 有序集
- 确认链路配置
- 准备进入 Configuration 状态

Configuration（配置）状态
^^^^^^^^^^^^^^^^^^^^^^^^^

协商链路宽度和通道编号。

**Configuration.Linkwidth.Start**
- 下游端口发送 TS1，提议链路宽度
- 上游端口响应

**Configuration.Linkwidth.Accept**
- 双方就链路宽度达成一致
- 确定使用的通道集

**Configuration.Lanenum.Accept/Wait**
- 分配通道编号
- 处理通道反转

**Configuration.Complete**
- 完成链路配置
- 准备进入 Recovery 或 L0

**Configuration.Idle**
- 发送空闲数据
- 等待进入 L0

Recovery（恢复）状态
^^^^^^^^^^^^^^^^^^^

从错误或速率变更中恢复链路。

**Recovery.RcvrLock**
- 重新建立位/块锁定
- 恢复链路同步

**Recovery.Equalization**
- 执行链路均衡
- 优化发送/接收设置

**Recovery.Speed**
- 协商数据速率变更
- 切换到新速率

**Recovery.RcvrCfg**
- 重新配置链路参数
- 确认配置一致

**Recovery.Idle**
- 稳定链路
- 准备返回 L0

L0 状态（正常工作）
^^^^^^^^^^^^^^^^^^^

链路全速工作状态：

- 正常 TLP 传输
- 流控和错误处理激活
- 可以进入低功耗状态

L0s 状态（待机）
^^^^^^^^^^^^^^^

轻量级低功耗状态，支持快速恢复。

**进入 L0s**
- 发送 EIOS（电气空闲有序集）
- 发送端进入电气空闲
- 接收端保持活动检测

**退出 L0s**
- 发送 FTS（快速训练序列）
- 接收端恢复锁定
- 典型恢复时间 < 1 μs

L1 状态（低功耗）
^^^^^^^^^^^^^^^^^

更深的低功耗状态，功耗更低但恢复更慢。

**进入 L1**
- 协议层请求
- 双方同意后进入
- 链路进入电气空闲

**退出 L1**
- 唤醒信号或请求
- 需要完整链路训练
- 恢复时间较长

L2 状态（辅助电源）
^^^^^^^^^^^^^^^^^^

最深的低功耗状态，仅保留辅助电源。

**特点**
- 主电源断开
- 仅保留唤醒检测电路
- 需要唤醒信号（WAKE#）或边带信号唤醒

Disabled 状态
^^^^^^^^^^^^^

链路被禁用：

- 响应软件禁用命令
- 接收端保持高阻态
- 可以重新启用

Loopback 状态（回环）
^^^^^^^^^^^^^^^^^^^^^

测试模式：

- 接收的数据回环发送
- 用于信号完整性测试
- 不处理 TLP

Hot Reset 状态
^^^^^^^^^^^^^^

热复位状态：

- 由下游端口发起
- 通过 TS1 有序集传播
- 重置链路状态但保持配置


电源管理
--------

电源管理是 PCIe 的重要特性，允许系统在空闲时降低功耗。

电源管理概述
~~~~~~~~~~~~

PCIe 电源管理分为两个层面：

**链路电源管理**
- 硬件自动控制
- 透明于软件
- 目标是降低链路功耗

**设备电源管理**
- 软件控制的设备状态
- 遵循 ACPI 规范
- 目标是降低整体设备功耗

链路状态电源管理
~~~~~~~~~~~~~~~~

PCIe 定义了多种链路电源状态，功耗从高到低递减。

L0 状态
^^^^^^^

全功耗工作状态：

- 链路完全激活
- 全速传输数据
- 所有电路正常工作

L0s 状态
^^^^^^^^

待机低功耗状态：

**特点**
- 单向进入（可以仅一端进入）
- 链路方向独立管理
- 恢复时间极短（< 1 μs）

**功耗节省**
- 关闭空闲方向的发送器
- 接收端保持最低活动检测
- 典型节省 10-30% 链路功耗

**适用场景**
- 短暂空闲期
- 不对称流量模式
- 频繁但短暂的业务间隔

L1 状态
^^^^^^^

深度低功耗状态：

**特点**
- 双向共同进入
- 需要双方协商同意
- 恢复时间较长（数十微秒）

**功耗节省**
- 双向都进入电气空闲
- 可以关闭更多电路
- 典型节省 50-80% 链路功耗

**进入条件**
- ASPM L1（自动）
- 软件请求（设备 D 状态转换）

L2 状态
^^^^^^^

辅助电源状态：

**特点**
- 主电源断开
- 仅 Vaux 供电
- 需要唤醒信号恢复

**功耗**
- 功耗极低（mW 级别）
- 仅保持唤醒检测

**恢复**
- 需要 WAKE# 信号
- 完整链路重新训练
- 恢复时间最长

活动状态电源管理（ASPM）
~~~~~~~~~~~~~~~~~~~~~~~~~

ASPM 是硬件自动管理的电源状态转换机制。

L0s ASPM
^^^^^^^^

**工作原理**

1. 发送方向检测到空闲
2. 等待 ASPM 定时器超时
3. 发送 EIOS 进入电气空闲
4. 有数据发送时发送 FTS 恢复

**配置**

- 通过链路控制寄存器启用
- 可以设置进入延迟阈值
- 可以针对不同方向独立配置

L1 ASPM
^^^^^^^

**工作原理**

1. 双方检测到链路空闲
2. 下游端口发送 PM_L1_ENTRY DLLP
3. 上游端口响应确认
4. 双方进入电气空闲

**前提条件**
- 上游设备支持 L1 ASPM
- 没有挂起的非发布请求
- 流控信用充足

**优势**
- 比 L0s 更大的功耗节省
- 恢复时间可接受
- 软件无需干预

L1 PM 子状态
~~~~~~~~~~~~

L1 PM 子状态提供更细粒度的功耗控制，用于 SRIS（独立参考时钟）场景。

L1.1 子状态
^^^^^^^^^^^

**特点**
- 关闭公共参考时钟检测
- 保持接收端终端阻抗
- 恢复速度较快

**适用场景**
- 使用独立参考时钟的系统
- 需要一定功耗节省但不要求最大

L1.2 子状态
^^^^^^^^^^^

**特点**
- 完全关闭链路电源
- 关闭接收端终端阻抗
- 最大功耗节省

**要求**
- 必须支持 SRIS
- 需要唤醒机制
- 恢复时间最长


系统架构
--------

系统架构涵盖中断处理、错误管理、虚拟通道、原子操作等高级功能。

中断和 PME 支持
~~~~~~~~~~~~~~~~

PCIe 使用消息信号中断取代传统引脚中断。

MSI（消息信号中断）
^^^^^^^^^^^^^^^^^^^

MSI 通过内存写入向 CPU 传递中断。

**工作原理**

1. 设备配置中断向量地址和数据
2. 设备产生中断时发送内存写 TLP
3. 地址指向中断控制器
4. 数据包含中断向量号

**MSI 能力**

- 支持 1、2、4、8、16 或 32 个向量
- 支持每向量屏蔽
- 支持 64 位地址

MSI-X
^^^^^

MSI-X 是 MSI 的增强版本：

**改进**

- 支持多达 2048 个向量
- 独立的向量地址和数据
- 更好的隔离和灵活性
- 支持 per-vector 屏蔽

**数据结构**

- **MSI-X Table**：向量地址、数据、控制
- **PBA (Pending Bit Array)**：待处理中断状态

错误信号和记录
~~~~~~~~~~~~~~

PCIe 定义了完整的错误检测、报告和记录机制。

错误分类
^^^^^^^^

**可纠正错误**

可以自动恢复的错误：

- TLP 传输错误（通过重传纠正）
- LCRC 错误
- 序列号错误
- 中毒 TLP 接收

**不可纠正错误**

无法自动恢复的错误：

- **非致命错误**：可能影响功能但不需要复位

  - 完成超时
  - 意外完成
  - 接收溢出

- **致命错误**：需要链路复位

  - 意预期的 TLP
  - 流控协议错误
  - 数据链路层协议错误

错误报告机制
^^^^^^^^^^^^

**完成状态码**

完成包携带状态码：

- **SC（Successful Completion）**：成功
- **UR（Unsupported Request）**：不支持请求
- **CA（Completer Abort）**：完成者中止

**错误消息**

设备发送错误消息通知根复合体：

- ERR_COR：可纠正错误消息
- ERR_NONFATAL：非致命错误消息
- ERR_FATAL：致命错误消息

高级错误报告（AER）
^^^^^^^^^^^^^^^^^^^

AER 提供详细的错误诊断信息：

**能力结构**

- 错误状态寄存器
- 错误屏蔽寄存器
- 错误严重性寄存器
- 头日志寄存器（记录错误 TLP）
- TLP 前缀日志

**优势**

- 精确定位错误源
- 记录错误上下文
- 支持软件恢复策略

下游端口遏制（DPC）
^^^^^^^^^^^^^^^^^^^

DPC 自动隔离下游错误，防止传播。

**工作原理**

1. 检测到致命错误
2. 自动阻塞下游端口
3. 阻止错误向上传播
4. 允许系统继续运行

**软件交互**

- 可以配置触发条件
- 支持 RP PIO（根端口编程 I/O）错误注入
- 需要软件参与恢复

虚拟通道支持
~~~~~~~~~~~~

虚拟通道（VC）允许在单条物理链路上承载多个独立的数据流。

VC 机制
^^^^^^^

**核心概念**

- **VC ID**：虚拟通道标识符（VC0-VC7）
- **TC 到 VC 映射**：流量类别到虚拟通道的映射
- **VC 仲裁**：多个 VC 之间的调度

**VC 仲裁**

- **严格优先级**：高优先级 VC 先服务
- **加权轮询（WRR）**：按权重轮转服务

**端口仲裁**

单个 VC 内多端口的调度，用于交换机。

应用场景
^^^^^^^^

- **等时流量**：音视频流保证带宽
- **控制流量**：管理流量与数据分离
- **QoS 保证**：不同服务质量需求

原子操作
~~~~~~~~

原子操作支持在设备内存上执行不可中断的操作。

支持的原子操作
^^^^^^^^^^^^^^

**FetchAdd**

.. code-block:: c

   atomic_fetch_add(address, value) {
       old_value = *address;
       *address = old_value + value;
       return old_value;
   }

**Unconditional Swap**

.. code-block:: c

   atomic_swap(address, new_value) {
       old_value = *address;
       *address = new_value;
       return old_value;
   }

**Compare and Swap (CAS)**

.. code-block:: c

   atomic_cas(address, compare_value, new_value) {
       old_value = *address;
       if (old_value == compare_value) {
           *address = new_value;
       }
       return old_value;
   }

应用价值
^^^^^^^^

- 无锁数据结构实现
- 多设备同步原语
- 分布式计数器
- 高效的消费者-生产者队列

访问控制服务（ACS）
~~~~~~~~~~~~~~~~~~~

ACS 提供对流量的精细控制，主要用于 SR-IOV 和多租户场景。

**功能**

- **源验证**：验证请求者身份
- **翻译阻塞**：阻止未经翻译的访问
- **P2P 控制**：控制对等流量路由
- **上游转发**：控制流量转发策略

**应用场景**

- 虚拟化隔离
- 安全域划分
- 多功能设备管理

精确时间测量（PTM）
~~~~~~~~~~~~~~~~~~~

PTM 提供分布式时间同步能力。

**工作原理**

1. PTM 请求者发送时间请求
2. PTM 响应者返回时间戳
3. 请求者计算路径延迟
4. 补偿后获得精确时间

**精度**

- 纳秒级时间精度
- 消除软件延迟影响
- 支持层级传播

**应用**

- 金融交易时间戳
- 工业控制同步
- 分布式测量系统


软件初始化和配置
----------------

软件通过配置空间发现和配置 PCIe 设备。

配置拓扑
~~~~~~~~

PCIe 继承了 PCI 的配置拓扑结构：

**配置地址**

每个功能通过三元组标识：

- Bus Number：总线号（0-255）
- Device Number：设备号（0-31）
- Function Number：功能号（0-7）

**配置空间**

每个功能有 4KB 配置空间：

- 前 256 字节：PCI 兼容配置空间
- 后 3840 字节：PCIe 扩展配置空间

配置机制
~~~~~~~~

PCI 兼容配置机制
^^^^^^^^^^^^^^^^

传统 x86 配置访问方式：

- 使用 I/O 端口 0xCF8/0xCFC
- 一次访问 1、2 或 4 字节
- 仅能访问前 256 字节配置空间

**访问流程**

1. 向 0xCF8 写入配置地址
2. 从 0xCFC 读取或写入数据

PCI Express 增强配置机制（ECAM）
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

ECAM 使用内存映射访问配置空间：

**地址映射**

.. code-block:: text

   | 位范围 | 字段 |
   |--------|------|
   | 63:28  | 固定基地址 |
   | 27:20  | Bus Number |
   | 19:15  | Device Number |
   | 14:12  | Function Number |
   | 11:2   | Register Offset |
   | 1:0    | 固定为 00 |

**优势**

- 完整访问 4KB 配置空间
- 更高效的访问方式
- 支持 64 位访问

配置寄存器类型
~~~~~~~~~~~~~~

PCI 兼容配置寄存器
^^^^^^^^^^^^^^^^^^

**Type 0/1 公共头**

- Vendor ID：厂商标识
- Device ID：设备标识
- Command：命令寄存器（启用内存/I/O 访问等）
- Status：状态寄存器
- Class Code：设备类型
- BAR：基地址寄存器

**Type 0 头（端点）**

- 6 个 BAR 用于地址映射
- 扩展 ROM 基地址
- 子系统 ID

**Type 1 头（桥接）**

- 主/次/从属总线号
- I/O 和内存范围
- 桥控制寄存器

PCI Express 能力结构
^^^^^^^^^^^^^^^^^^^^

PCIe 能力结构位于配置空间，包含：

- **能力寄存器**：设备/链路能力
- **控制寄存器**：功能启用/禁用
- **状态寄存器**：当前状态

关键寄存器：

- Link Capabilities：链路速度、宽度支持
- Link Control：链路电源管理控制
- Link Status：当前链路状态
- Device Capabilities：设备特性
- Device Control：错误报告控制

PCI Express 扩展能力
^^^^^^^^^^^^^^^^^^^^

扩展能力位于 256B 之后的配置空间：

- 高级错误报告（AER）
- 虚拟通道（VC）
- 设备序列号
- 电源预算
- SR-IOV
- ATS
- 其他


电气子块
--------

电气规范定义了发送器、接收器和通道的电气特性。

电气规范概述
~~~~~~~~~~~~

电气规范确保不同厂商设备的互操作性。

**关键参数**

- 电压电平
- 时序参数
- 阻抗匹配
- 抖动容限

发送器规范
~~~~~~~~~~

电压参数
^^^^^^^^

**差分峰峰值电压**

.. list-table::
   :header-rows: 1
   :widths: 20 30 50

   * - 速率
     - Vtx-diff-pp
     - 说明

   * - 2.5 GT/s
     - 0.8-1.2 V
     - 全摆幅模式

   * - 5.0 GT/s
     - 0.8-1.2 V
     - 全摆幅模式

   * - 8.0+ GT/s
     - 由均衡设置决定
     - 可调整范围

**去加重**

高速信号需要去加重补偿通道损耗：

- 降低连续相同位的幅度
- 增强跳变边沿
- 典型设置：-3.5 dB 或 -6 dB

抖动参数
^^^^^^^^

抖动分为多个成分：

- **随机抖动（RJ）**：高斯分布，无法预测
- **确定性抖动（DJ）**：有界，由特定原因引起
- **数据相关抖动（DDJ）**：与数据模式相关
- **周期性抖动（PJ）**：周期性干扰

总抖动预算由规范定义，必须满足才能保证误码率要求。

接收器规范
~~~~~~~~~~

接收器需要满足应力眼图要求。

**应力眼图**

接收器必须能正确接收经过损伤链路的信号：

- 眼图高度要求：最小可检测电压
- 眼图宽度要求：最小定时裕量

**均衡要求**

高速接收器需要均衡器：

- CTLE：连续时间线性均衡
- DFE：判决反馈均衡

通道容差
~~~~~~~~

**插入损耗**

通道的插入损耗影响信号完整性：

- 低速率：损耗要求宽松
- 高速率：严格限制最大损耗

**串扰**

近端串扰（NEXT）和远端串扰（FEXT）必须控制：

- 差分对间耦合最小化
- 布局优化减少串扰源


单根 I/O 虚拟化（SR-IOV）
-------------------------

SR-IOV 允许单个物理设备呈现为多个虚拟功能。

SR-IOV 架构概述
~~~~~~~~~~~~~~~

SR-IOV 定义了两种功能类型：

**物理功能（PF）**

- 完整的配置能力
- 管理 VF 的创建和配置
- 处理所有 SR-IOV 控制操作

**虚拟功能（VF）**

- 轻量级功能实例
- 独立的配置空间和工作资源
- 共享物理设备的底层硬件

虚拟化优势
^^^^^^^^^^

- **直接设备分配**：虚拟机直接访问硬件
- **减少软件开销**：绕过虚拟机管理程序
- **I/O 性能提升**：接近原生性能

初始化和资源分配
~~~~~~~~~~~~~~~~

VF 发现和配置
^^^^^^^^^^^^^

**枚举流程**

1. 发现 PF，读取 SR-IOV 能力
2. 配置 VF 数量（NumVFs）
3. 配置 VF BAR
4. 启用 VF（VF Enable）
5. 枚举 VF 作为独立功能

**资源分配**

- VF BAR：从 PF 的 VF BAR 模板派生
- 中断：每个 VF 可配置独立 MSI/MSI-X
- DMA：独立的地址空间

SR-IOV 配置
~~~~~~~~~~~

SR-IOV 扩展能力结构包含：

- TotalVFs：支持的最大 VF 数量
- NumVFs：当前启用的 VF 数量
- VF BAR：VF 的基地址寄存器
- First VF Offset：第一个 VF 的功能号
- VF Stride：VF 之间的功能号间隔

错误处理
~~~~~~~~

VF 错误独立报告：

- 每个 VF 有独立的错误状态
- 可以独立屏蔽和启用错误报告
- 错误消息携带 VF 的请求者 ID


地址翻译服务（ATS）
-------------------

ATS 允许设备缓存翻译后的地址，减少 IOMMU 访问开销。

ATS 架构概述
~~~~~~~~~~~~~

ATS 解决的问题：

- 设备 DMA 访问需要虚拟地址翻译
- 每次访问 IOMMU 翻译增加延迟
- 缓存翻译结果可提高性能

ATS 工作流程
^^^^^^^^^^^^

1. 设备发送翻译请求（未翻译地址）
2. IOMMU 翻译地址并返回结果
3. 设备缓存翻译结果
4. 后续访问使用缓存的翻译地址

翻译服务
~~~~~~~~

翻译请求
^^^^^^^^

请求格式：

- 未翻译地址（48 位或 64 位）
- 读/写权限请求
- 页面大小请求
- PASID（可选，用于进程地址空间）

翻译完成
^^^^^^^^

响应格式：

- 翻译后的地址
- 页面大小（可大于请求）
- 权限标志（R/W/X）
- 全局映射标志

页面请求服务
~~~~~~~~~~~~

当设备访问未翻译或无权限的页面时，可以请求页面驻留。

**页面请求消息**

- 请求类型：缺页/权限请求
- PASID：进程标识
- 地址：需要驻留的页面
- 请求索引：匹配响应

**页面响应消息**

- 成功/失败状态
- 无效 PASID 标志
- 请求索引匹配

ATS 配置
~~~~~~~~

ATS 扩展能力结构：

- ATS 能力：支持的队列深度
- ATS 控制：启用/禁用、地址翻译使能
- 状态：当前操作状态
