Linux 内核源码分析 — 提示词编写指南
========================================

本文档提供一套结构化的提示词模板，用于指导 AI 分析 Linux 内核源码。
核心原则：**上下文越精确，分析越深入**。

1. 基本分析模板
================

分析单个文件时使用：

.. code-block:: text

   请分析以下 Linux 内核源码文件，输出中文：

   文件：src/linux-6.1-rc6/drivers/xxx/yyy.c

   要求：
   1. 驱动功能概述（一句话描述 + 所属子系统）
   2. 核心数据结构及其关系
   3. 初始化/注册流程（从入口点到硬件可用）
   4. 操作函数集（回调表：函数名 → 职责 → 调用者）
   5. 中断处理流程（中断类型 → 触发条件 → 处理链）
   6. 并发保护机制（锁类型 → 保护对象 → 无锁设计）
   7. CPU 热插拔/电源管理处理
   8. 与用户空间的接口（sysfs/ioctl/proc 等）
   9. 本文件是框架层还是具体实现？如果是框架层，列出需要具体驱动实现的回调

2. 跨文件追踪模板
==================

分析一个功能涉及多个文件时使用：

.. code-block:: text

   请分析 Linux 内核中 <功能名称> 的完整实现。

   已知入口文件：src/linux-6.1-rc6/path/to/entry.c

   要求：
   1. 从入口函数开始，画出完整的调用链（函数名 + 文件:行号）
   2. 识别所有涉及的数据结构（struct 定义位置 + 成员说明）
   3. 按执行阶段分组：初始化阶段 / 运行时阶段 / 清理阶段
   4. 标注关键的内核 API 调用及其作用
   5. 画出组件交互图（用 ASCII 或文字描述）

3. 机制对比模板
================

对比两种实现机制时使用：

.. code-block:: text

   请对比 Linux 内核中 <机制A> 和 <机制B> 的实现差异：

   文件A：src/linux-6.1-rc6/path/to/a.c
   文件B：src/linux-6.1-rc6/path/to/b.c

   要求：
   1. 两者解决的问题分别是什么？
   2. 数据结构差异
   3. 注册/初始化流程差异
   4. 性能特征差异（锁粒度、内存分配、中断上下文）
   5. 各自的适用场景
   6. 用表格对比

4. 数据流追踪模板
==================

追踪一个数据从产生到消费的完整路径：

.. code-block:: text

   请追踪 Linux 内核中 <数据/事件> 从产生到消费的完整路径。

   起点：src/linux-6.1-rc6/path/to/producer.c
   终点：src/linux-6.1-rc6/path/to/consumer.c

   要求：
   1. 数据结构定义（在各层之间的变换）
   2. 完整传递路径（函数调用链 + 文件:行号）
   3. 每一层的处理逻辑（做了什么变换/过滤/聚合）
   4. 并发安全如何保证
   5. 错误处理路径（数据丢失/重试/回退）

5. 硬件交互分析模板
====================

分析驱动与硬件寄存器的交互：

.. code-block:: text

   请分析以下驱动与硬件的交互方式：

   文件：src/linux-6.1-rc6/drivers/xxx/yyy.c
   参考手册：<手册名称或规范文档路径>

   要求：
   1. 硬件寄存器列表（偏移 + 名称 + 读写属性）
   2. 寄存器操作封装函数（readx/writex 调用位置）
   3. 初始化时的寄存器配置序列
   4. 运行时的寄存器读写时机（中断中？进程上下文？）
   5. 与硬件手册的对应关系（代码中的操作对应手册哪一节）

6. 提示词编写原则
==================

6.1 指定文件路径
-----------------

.. list-table::
   :header-rows: 1
   :widths: 50 50

   * - 差
     - 好
   * - 分析一下 PMU 驱动
     - 分析 src/linux-6.1-rc6/drivers/perf/arm_pmu.c
   * - 看看中断处理部分
     - 分析 arm_pmu.c 中 armpmu_dispatch_irq() 到 pmu->handle_irq() 的中断处理链

6.2 指定分析深度
-----------------

.. list-table::
   :header-rows: 1
   :widths: 50 50

   * - 差
     - 好
   * - 解释这个函数
     - 用 3 句话解释这个函数，并标注每一行代码的作用
   * - 分析这个驱动
     - 画出从 subsys_initcall 到 perf_pmu_register 的完整注册流程图

6.3 提供已知上下文
-------------------

.. list-table::
   :header-rows: 1
   :widths: 50 50

   * - 差
     - 好
   * - 为什么这里要 reset？
     - 在 CPU_PM_EXIT（休眠唤醒）时为什么必须先调用 pmu->reset()？考虑 PMU 硬件在掉电后寄存器状态 UNKNOWN 的因素
   * - 这个锁有什么用？
     - armpmu_event_update() 为什么用 local64_cmpxchg 而不是 spinlock？考虑中断上下文的约束

6.4 要求结构化输出
-------------------

.. list-table::
   :header-rows: 1
   :widths: 50 50

   * - 差
     - 好
   * - 帮我总结一下
     - 用表格列出所有回调函数：函数名 | 文件:行号 | 职责 | 调用者
   * - 这些数据结构什么关系？
     - 用 ASCII 图画出 struct arm_pmu、struct pmu_hw_events、struct pmu 之间的关系

6.5 指定输出语言和格式
------------------------

.. list-table::
   :header-rows: 1
   :widths: 50 50

   * - 差
     - 好
   * - （无语言要求）
     - 输出中文，技术术语保留英文原文
   * - （无格式要求）
     - 输出为 Markdown，代码块标注语言，表格对比

7. 常用追问模板
================

深入分析时的追问：

.. code-block:: text

   # 追问设计决策
   为什么这里选择 <方案A> 而不是 <方案B>？考虑了哪些 trade-off？

   # 追问边界条件
   这个函数在以下场景的行为是什么：
   1. 多个 CPU 同时调用
   2. 中断上下文中被调用
   3. 传入参数为 NULL/0
   4. 资源已耗尽

   # 追问历史演进
   这个实现从哪个内核版本开始引入？解决了什么问题？
   查看 git log: git log --oneline src/linux-6.1-rc6/path/to/file.c

   # 追问与规范的对应
   这段代码对应 <ARM 规范文档> 的哪一节？实现是否完整？

   # 追问性能影响
   这个函数在热路径上被调用吗？是否有锁竞争或 cache line bouncing 的风险？

8. 实战示例
============

以 arm_pmu.c 为例，完整的提问序列：

.. code-block:: text

   # 第一轮：整体理解
   分析 src/linux-6.1-rc6/drivers/perf/arm_pmu.c，输出：
   1. 驱动功能概述
   2. 核心数据结构
   3. 注册流程（从入口到 perf_pmu_register）
   4. pmu 回调函数表（函数名 → 职责）
   5. 中断处理链
   6. CPU 热插拔处理
   7. 框架层 vs 具体驱动的接口边界

   # 第二轮：深入数据流
   从用户执行 `perf stat -e cycles` 开始，追踪代码路径：
   perf_event_open() → event_init → add → start → 溢出中断 → stop → del
   每一步标注文件:行号和关键操作

   # 第三轮：并发分析
   分析 arm_pmu 中的并发保护：
   1. 哪些数据被多个 CPU 并发访问？
   2. 用了什么同步机制？为什么选这个机制？
   3. armpmu_event_update() 中 local64_cmpxchg 的 CAS 循环为什么需要重试？

   # 第四轮：与硬件规范对应
   arm_pmu.c 中的以下操作对应 ARM PMUv3 规范的哪些寄存器：
   - enable/disable_event
   - read_counter/write_counter
   - set_event_filter
   - reset
