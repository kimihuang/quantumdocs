CXL 3.0 协议栈深度分析：CXL.io、CXL.cache 与 CXL.mem
========================================================

本文档基于 CXL 3.0 Specification version 0.7 (December 2021) 第 3 章 Transaction Layer 内容，
深入分析 CXL.io、CXL.cache 和 CXL.mem 三种核心协议的架构、通道定义、事务流程和关键链路。

参考文档：``docs/pdf/CXL/CXL_Spec_3.0_v0.7.pdf.md``

   :depth: 3
   :local:

协议概述与分层架构
==================

CXL 事务层定位
--------------

CXL 事务层位于 Flex Bus 分层架构的顶层，负责定义设备与主机之间的通信协议。Figure 3-1 展示了 CXL.io 事务层在 Flex Bus 分层中的位置 [Section 3.1]_。

**三层协议架构**:

- **CXL.io**: 基于 PCIe 的非相干 I/O 协议，用于设备发现、枚举、配置、中断和 DMA 操作
- **CXL.cache**: 相干性协议，使设备缓存与主机内存保持 MESI 相干性
- **CXL.mem**: 内存访问协议，允许主机访问设备附加内存 (HDM)

CXL 3.0 关键增强 [Section 3 引言]_
-----------------------------------

- **256B Flit 支持**: 与 PCIe 6.0 对齐，消息头尺寸调整以匹配 Flit 结构
- **68B Flit 保持不变**: 支持 2-bit CacheID 用于设备扩展、Back-Invalidation Snoop 和对称内存流
- **CXL.cache 去偏置翻转**: 弃用 "Bias Flip" 流，改用 CXL.mem Back-Invalidation Snoop 通道处理
- **CXL.io 无序 I/O**: 支持直接 Peer-to-peer 访问 HDM
- **CXL.mem 新增通道**: Back-Invalidation Snoop (BISnp) 和对称内存流，支持 Type 2 设备实现包含式 Snoop Filter

CXL.io 协议详解
===============

协议特性与端点行为 [Section 3.1]_
----------------------------------

CXL.io 提供非相干的 load/store 接口用于 I/O 设备。事务类型、包格式、基于信用的流量控制和虚拟通道管理遵循 PCIe 定义。

**CXL.io 端点要求**:

- 当链路配置为 RCiEP 模式 (CXL 1.1 模式) 时，必须暴露为 PCIe RCiEP
- 否则暴露为 PCIe Endpoint
- 参与 CXL 协议的端点函数不得生成 INTx 消息
- MLD 组件的所有 CXL.io 端点函数 (包括非 CXL 函数) 均不允许生成 INTx 消息

CXL 电源管理 VDM 消息格式 [Section 3.1.2]_
-----------------------------------------

CXL 电源管理消息作为 PCIe Vendor Defined Type 0 消息发送，带有 4-DWORD 数据负载。包括 PMREQ、PMRSP 和 PMGO 消息。

**消息特征** [Figure 3-2]_:

- Fmt 和 Type 字段设置为带数据的消息
- 所有消息使用 "Local-Terminate at Receiver" 路由
- Message Code 设置为 Vendor Defined Type 0
- Vendor ID 字段设置为 1E98h
- 字节 15 包含 VDM Code，设置为 "CXL PM Message" (68h)
- 4-DWORD 数据负载包含 CXL PM 逻辑操作码和相关信息

**数据负载字段定义** [Table 3-1]_:

+------------------------+-----------+------------------------------------------+
| 字段                    | 位宽       | 描述                                      |
+========================+===========+==========================================+
| PM Logical Opcode[7:0] | 8 bits    | 电源管理命令编码：                        |
|                        |           | - 00h: AGENT_INFO                         |
|                        |           | - 02h: RESETPREP                          |
|                        |           | - 04h: PMREQ (PMRSP 和 PMGO)              |
|                        |           | - 06h: Global Persistent Flush (GPF)      |
|                        |           | - FEh: CREDIT_RTN                         |
+------------------------+-----------+------------------------------------------+
| PM Agent ID[6:0]       | 7 bits    | IP2PM: 分配给设备的 PM agent ID            |
|                        |           | Host 通过首个 CREDIT_RTN 消息的             |
|                        |           | TARGET_AGENT_ID 字段传达                   |
+------------------------+-----------+------------------------------------------+
| Parameter[15:0]        | 16 bits   | 依不同命令有不同含义                       |
+------------------------+-----------+------------------------------------------+

信用和 PM 初始化流程 [Section 3.1.2.1]_
--------------------------------------

PM 信用和初始化过程是链路本地的。Figure 3-3 展示了使用 PM2IP.CREDIT_RTN 和 PM2IP.AGENT_INFO 消息初始化电源管理协议。

**初始化规则**:

1. CXL Upstream Port PMU 必须能够接收和处理 CREDIT_RTN 消息，不依赖于任何其他 PM2IP 消息
2. CREDIT_RTN 消息不使用信用
3. 首个 CREDIT_RTN 消息中，NUM_CREDITS 字段表示发起方可以接收的依赖信用的 PM 消息数量
4. 后续 CREDIT_RTN 消息中，NUM_CREDITS 字段表示自上次消息以来释放的 PM 信用数量
5. Downstream Port PMU 通过 TARGET_AGENT_ID 字段分配 PM_AGENT_ID 给 Upstream Port PMU
6. Upstream Port PMU 必须等待来自 Downstream Port PMU 的 CREDIT_RTN 消息后才能发起任何 IP2PM 消息

**Upstream Port PMU 规则**:

- 必须等待接收 PM2IP.CREDIT_RTN 消息后才能发起 IP2PM 消息
- 必须从首个接收的 PM2IP 消息中提取 TARGET_AGENT_ID 并用作其 PM_AGENT_ID
- 必须实现足够资源来接收和处理任何 CREDIT_RTN 消息
- 必须至少实现一个信用来接收 PM2IP 消息
- 必须尽快返回信用以防止阻塞
- 建议扣留信用不超过 10 微秒

错误传播 [Section 3.1.5]_
--------------------------

CXL.cache 和 CXL.mem 错误由设备检测后，通过 CXL.io 流量向上游传播。这些错误记录在 PCIe AER 寄存器中作为可纠正和不可纠正的内部错误。

ATS 上的内存类型指示 [Section 3.1.6]_
--------------------------------------

某些内存区域的请求只能通过 CXL.io 发出，不能使用 CXL.cache。主机通过 ATS 完成消息中的 Issue-on-CXL.io 位向设备指示这些区域。

**ATS 请求格式** [Figure 3-5]_:

- 64-bit: DWORD3, Byte 3, Bit 3
- 32-bit: DWORD2, Byte 3, Bit 3
- 0: 表示请求由不支持内存类型指示的函数发起
- 1: 表示请求由支持内存类型指示的函数发起 (所有 CXL 设备函数必须设置此位)

**ATS 翻译完成数据条目** [Figure 3-6]_:

- DWORD1, Byte 2, Bit 1:
- 0: 页面请求可在所有 CXL 协议上发出
- 1: 函数只能在 CXL.io 上发出页面请求，使用 CXL.cache 协议属于违规

CXL.cache 协议详解
==================

通道架构概览 [Section 3.2.1]_
------------------------------

CXL.cache 协议定义了设备与主机之间的大量请求、响应和数据传输交互。接口由每个方向的三个通道组成：Request、Response 和 Data。

**六个独立通道** [Figure 3-7]_:

**Device to Host (D2H) 方向**:

- **D2H Request**: 携带设备到主机的新请求，主要访问内存
- **D2H Response**: 携带所有设备到主机的响应
- **D2H Data**: 携带所有设备和字节使能到主机

**Host to Device (H2D) 方向**:

- **H2D Request**: 携带主机到设备的请求 (snoop)
- **H2D Response**: 携带排序消息和写数据拉取
- **H2D Data**: 传递设备读请求的数据

通道特性:

- 独立通道允许不同类型的消息使用专用线路，实现解耦和更高的有效吞吐量
- D2H Request 可以被反压而不影响其他通道
- D2H Data 必须取得进展，否则可能导致死锁
- H2D Requests 可能因设备资源不足而被反压，但必须在无需 D2H Requests 进展的情况下释放资源

通道信用机制 [Section 3.2.2.2]_
--------------------------------

每个通道必须使用信用发送任何消息并从接收方收集信用返回。接收方在处理消息后返回信用。

**通道信用表** [Table 3-4]_:

设备到主机请求语义 [Section 3.2.4.2]_
------------------------------------

四种语义分类:

1. **CXL.cache Read**: 完整缓存行读取
2. **CXL.cache Read0**: 无数据传输的读取
3. **CXL.cache Read0/Write**: 读零写语义
4. **CXL.cache Write**: 写入语义

CXL.cache Read 行为 [Figure 3-8]_
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

::

   Device                           Host
   ------                           ----
   D2H Read Request Message  --->
   
   <---  H2D Response (GO) Message (RdCurr 不存在)
   
   <---  H2D Data Message (32 B)
   <---  H2D Data Message (32 B)

**特性**:

- 必须有 D2H request 信用并在 D2H CXL.cache request 通道发送请求消息
- 需要零个或一个响应 (GO) 消息和总计 64 字节数据消息
- 响应和数据消息都针对初始 D2H 请求包的 CQID 字段指向的设备追踪条目
- 设备条目必须保持活动直到收到所有主机消息
- 设备必须有保留数据缓冲区以在发送请求后立即接收全部 64 字节数据
- 收到响应和数据消息后，事务可视为完成并释放条目

CXL.cache Read0 行为 [Figure 3-9]_
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

::

   Device                           Host
   ------                           ----
   D2H Read0 Request Message  --->
   
   <---  H2D Response (GO) Message

**特性**:

- 必须有 D2H request 信用并发送消息
- 接收响应消息但无数据消息
- 收到 GO 消息后可视为完成
- 大多数特殊周期 (如 CLFlush) 和其他杂项请求属于此类

**Write 语义特性**:

- 必须有 D2H request 信用
- 主机接收到请求后，必须发送两个分离或一个合并的 GO-I 和 WritePull 消息
- GO 消息绝不能在 WritePull 之前到达，但可以同时到达
- 收到 GO-I 后，从内存排序和缓存相干性角度认为存储完成
- WritePull 触发设备发送总计恰好 64 字节的数据消息
- 收到 GO-I 并发送所需数据消息后，设备认为事务完成

**Buried Cache State 规则** [Section 3.2.5.14]:

Buried Cache state 指设备发起 CXL.cache 协议新请求时，设备相干引擎 (DCOH) 中注册的缓存行状态。

- 设备不得为处于 Modified、Exclusive 或 Shared 状态的缓存行发起 Read
- 设备不得为处于 Modified 或 Exclusive 状态的缓存行发起 RdOwnNoData
- 设备不得为处于 Modified、Exclusive 或 Shared 状态的缓存行发起 Read0-Write
- 所有 *Evict 操作码必须遵守适用用例
- CacheFlushed 操作码不针对特定缓存行，设备在任何缓存行处于 Modified、Exclusive 或 Shared 状态时不得发出

**一般假设** [Section 3.2.5.13]:

1. 主机不会保留设备交付的 CXL.cache 请求的顺序，设备必须维护请求顺序
2. 主机选择的顺序对读取和写入有不同的传达方式
3. 设备可以缓存所有权并在内部排序写入
4. 对于设备读取，主机用 GO 消息传达缓存行所有权
5. 主机不得为已发送数据响应但未发送 GO 的地址发送 snoop
6. 除 Evict 外的写请求永远不会用标记为 Bogus 的数据响应 WritePulls
7. 主机不得向同一设备请求发送两个缓存行数据响应
8. 对于给定事务，H2D 数据传输必须在连续包中以自然顺序进行
9. 缓存行的 D2H 数据传输必须在连续包中进行，按自然块顺序
10. 设备 snoop 响应不得依赖于任何其他通道或请求
11. 主机在收集先前 snoop 的所有响应和数据前不得发送第二个 snoop 请求
12. H2D Response 和 H2D Data 消息必须在不需任何其他事务进展的情况下排出
13. 主机不得返回 GO-M 给实际上相对于内存未修改的数据
14. 主机不得将未修改的数据写回内存
15. 除 WOWrInv 和 WOWrInvF 外，所有其他写入都是强序的

CXL.mem协议详解
================

协议概述 [Section 3.3.1]_
--------------------------

CXL 内存协议 (CXL.mem) 是 CPU 和内存之间的事务接口。协议可用于多种不同的内存附加选项，包括内存控制器位于主机 CPU 内、加速器设备内或内存缓冲芯片中。

**主从架构**:

- **CXL.mem Master**: 负责发起 CXL.mem 请求 (如读取、写入等) 的代理 (CPU 相干引擎)
- **CXL.mem Subordinate**: 负责响应 CXL.mem 请求 (如数据、完成等) 的代理 (内存设备)

**事务方向**:

- **M2S (Master to Subordinate)**: 主机到设备的请求
- **S2M (Subordinate to Master)**: 设备到主机的响应

**M2S 消息类别**:

- **Request without data (Req)**: 无数据请求
- **Request with Data (RwD)**: 带数据请求
- **Back-Invalidation Response (BIRsp)**: 反向无效响应

**S2M 消息类别**:

- **Response without data (NDR)**: 无数据响应
- **Response with data (DRS)**: 带数据响应
- **Back-Invalidation Snoop (BISnp)**: 反向无效 Snoop

通道架构 [Section 3.3.2]_
--------------------------

**设备接口的 6 个通道** [Figure 3-15]_:

**M2S 方向 (主机到设备)**:

- **M2S Req**: 请求无数据
- **M2S RwD**: 请求带数据
- **M2S BIRsp**: Back-Invalidation 响应

**S2M 方向 (设备到主机)**:

- **S2M NDR**: 响应无数据
- **S2M DRS**: 响应带数据
- **S2M BISnp**: Back-Invalidation Snoop

**Type 2 设备支持**:

- Type 2 设备必须支持 BI* 通道 (S2M BISnp 或 M2S BIRsp)，这是 CXL 3.0 引入的
- CXL 3.0 之前设计的 Type 2 设备不会有 BI* 通道
- 支持直接 peer-to-peer 访问 HDM 的 Type 3 设备将支持 BI* 通道

**通道独立性**:

一般而言，CXL.mem 通道独立工作以确保维持进展。通道间的具体排序允许和要求详见 Section 3.4 和 Section 3.5。在通道内没有排序规则，但例外情况见 Section 3.3.11。

设备缩放 [Section 2.5]_
------------------------

CXL 支持在 VH 下连接最多 16 个 Type 1 和/或 Type 2 设备。为支持此缩放，Type 2 设备需要使用 CXL.mem协议中的 BISnp 通道来管理 HDM 区域的相干性。

CXL 3.0 中引入的 BISnp 通道替代了使用 CXL.cache 协议管理设备 HDM 区域的相干性。使用 CXL.cache 进行 HDM 相干性管理的 Type 2 设备限制为每个主机桥一个设备。

Fabric 架构 [Section 7.7]_
==========================

Fabric 概述
-----------

CXL 3.0 添加了新功能以从节点扩展到机架级互连。主要变更领域包括 [Section 7.7]_:

- 使用 Port Based Routing 和 12 位端口标识符扩展 CXL fabric 规模
- Fabric RAS 扩展用于错误遏制和报告
- Fabric Management - 允许管理系统管理 CXL 互连
- 安全 - 允许多个 EP 之间安全通道的扩展
- Hot Plug/Remove 支持 - 允许资源的动态管理
- Host-to-Host 通信 - 允许主机之间更低延迟的通信
- Device to Device 通信 - 更低延迟的设备到设备通信

**Fabric 规模**:

- 支持跨越 4096 个 CXL/PCIe 端口的 fabric
- 使用 12 位源端口标识符 (SPID) 和 12 位目标端口标识符 (DPID)

高层 Fabric 图 [Figure 7-24]_
------------------------------

Fabric 包含以下组件:

- **n 个 Switch Edge Ports (SEPx)**: 每个边缘端口可连接到 CXL Root Port (RP) 或 CXL/PCIe 设备 (Dev)
- **Fabric Manager (FM)**: 通过管理网络连接到 CXL Fabric 和所有端点
- **管理网络**: 可以是简单的 2 线接口 (如 SMBus、I2C、I3C 或 SPI) 或复杂的以太网

**系统域**:

- 同一系统域内的 Root Ports 可以与附加的 CXL 设备保持相干性
- 不同系统域的 Root Ports 不要求硬件相干性
- MLD 和 GFAM 设备可以支持跨多个系统域的硬件管理缓存相干性
- 不支持域内或跨域的设备到设备硬件缓存相干性

**三级保护层次** [Figure 7-32]_:

1. **第一级**: 主机或请求设备的页表
2. **第二级**: 交换机入口端口，编程 Fabric 路由的位置
3. **第三级**: 目标设备本身的细粒度保护

**GFAM 设备保护机制** [Figure 7-33]_:

GFAM 设备的地址空间划分为内存地址范围块，每块分配一个 Group ID (GrpID)。

**表结构**:

- **Group Table RAM (GTR)**: 划分整个 GFAM 设备地址空间为 64 MB 或更大的等幂二次大小块
- **Group Table CAM (GTC)**: 用于定义各种大小的自然对齐幂二次地址范围并分配 GrpID (可选)
- **SPID Access Table (SAT)**: 使用请求 SPID 索引产生向量，识别 SPID 允许访问哪些内存组 (GrpAccVec)
- **Group Attributes Table (GAT)**: GrpID 索引提供组属性，如是否使用硬件管理或软件管理缓存相干性

**访问流程**:

1. 请求地址经过掩码和移位形成 GTR 索引，产生 GrpID
2. GTC 命中优先于 GTR 查找
3. 并行地，请求 SPID 索引 SAT 产生 GrpAccVec
4. GrpID 用于选择 GrpAccVec 位确定是否允许访问
5. GrpID 还可索引 GAT 提供组属性

PBR 交换机对 HBR 交换机的支持 [Section 7.7.4]_
--------------------------------------------------

**两种交换机类型**:

- **HBR (Hierarchy Based Routing)**: 使用 4 位 LD-ID 和 CacheID
- **PBR (Port Based Routing)**: 使用 12 位 SPID 和 DPID

**消息转换**:

PBR 交换机将来自 HBR 主机、设备和交换机的消息转换为 PBR 消息格式以便在 Fabric 上路由。

**CXL.io 转换** [Section 7.7.4.1.1]:

- **入口 (上游) 端口**:
  - 交换机将 TLP 映射到 CXL.io 消息 (仅 PCIe 设备)
  - 交换机添加 CXL.io 前缀到消息
  - 前缀包含 12 位 SPID 和 12 位 DPID

- **出口 (下游) 端口**:
  - 交换机从消息中删除 TLP 前缀
  - 交换机将 CXL.io 消息映射到 PCIe TLP (仅 PCIe 设备)

**CXL.cache 转换** [Section 7.7.4.1.2]:

**D2H Req**:

- **入口 (下游) 端口**:
  - 交换机添加 12 位 SPID 字段
  - 交换机添加 12 位 DPID 字段 (基于 VCS 绑定)
  - 交换机从消息中删除 4 位 CacheID 字段

- **出口 (上游) 端口**:
  - 交换机将 12 位 SPID 映射到 4 位 CacheID 字段并添加到消息
  - 交换机从消息中删除 12 位 SPID 和 12 位 DPID 字段

**CXL.mem 转换** [Section 7.7.4.1.3]:

**M2S Req 和 M2S RwD 消息**:

- **入口 (上游) 端口**:
  - 交换机添加 12 位 SPID 字段 (FM 分配的值)
  - 交换机添加 12 位 DPID 字段 (基于 HDM 解码器)
  - 交换机从消息中删除 4 位 LD-ID 字段

- **出口 (下游) 端口**:
  - 对于 SLD 设备，交换机插入值为 0 的 4 位 LD-ID 字段

Fabric 管理应用编程接口 [Section 7.6]_
=========================================

Fabric Manager 模型
--------------------

**Fabric Manager (FM)**: 外部逻辑进程，使用 FM API 命令集查询和配置系统的运行状态 [Section 7.6.1]_。

**FM 形式**:

- 运行在主机上的软件
- 运行在 BMC 上的嵌入式软件
- 运行在另一个 CXL 设备或 CXL 交换机上的嵌入式固件
- CXL 设备内部的状态机

**CCI (Component Command Interface)**:

FM 通过 CCI 发送命令，CCI 通过设备的邮箱寄存器或 MCTP 兼容接口暴露 [Section 7.6.2]_。

**示例 Fabric 管理模型** [Figure 7-18]_:

::

   VCS 0        VCS 1
   Virtual to Physical Binding
   
   BMC (MCTP EP)
   |
   +-- PCIe or CXL Device
   |
   +-- Type 3 Pooled Device (PPB, LD FFFF, LD 0, LD 1)
   |
   +-- CXL Switch (vPPB, PHY PORT)
       |
       +-- Type 3 Pooled MLD (LD FFFF, LD 0, LD 1)
   
   Fabric Manager (FM API Support)

**连接示例**:

- FM 通过任何 MCTP 兼容互连直接连接到 CXL 设备可直接发布 FM 命令
- FM 直接连接到 CXL 交换机可使用交换机隧道传输 FM 命令到直接附加的 MLD 组件
- FM 可嵌入 CXL 设备内部

关键链路流程总结
==================

CXL.io 关键流程
----------------

1. **电源管理初始化**:
   - 使用 CREDIT_RTN 和 AGENT_INFO 消息建立信用
   - Downstream Port 分配 PM_AGENT_ID 给 Upstream Port
   - Upstream Port 必须等待 CREDIT_RTN 后才能发起 IP2PM 消息

2. **错误传播**:
   - CXL.cache 和 CXL.mem 错误通过 CXL.io 向上传播
   - 记录在 PCIe AER 寄存器中

3. **ATS 内存类型指示**:
   - 主机通过 Issue-on-CXL.io 位指示某些页面只能在 CXL.io 上访问
   - 设备违反此限制使用 CXL.cache 属于违规

CXL.cache 关键流程
-------------------

1. **Read 事务流程**:
   ::
   
      Device --D2H Read Request--> Host
      Device <--H2D Response (GO)-- Host (可选，RdCurr 不存在)
      Device <--H2D Data (32B x2)- Host

2. **Write 事务流程** (Posted):
   ::
   
      Device --D2H Write Request--> Host
      Device <--H2D GO+WritePull----- Host
      Device --D2H Data (64B)-----> Host

3. **Write 事务流程** (Non-posted):
   ::
   
      Device --D2H Write Request--> Host
      Device <--H2D Write Pull----- Host
      Device --D2H Data (64B)-----> Host
      Device <--H2D GO------------- Host

4. **Snoop 事务流程**:
   ::
   
      Host --H2D Snoop Request--> Device
      Host <--D2H Response (Rsp-X)- Device
      Host <--D2H Data (如果需要)-- Device (如果 Rsp*Fwd*)

5. **Evict 事务流程**:
   ::
   
      Device --D2H Evict Request--> Host
      Device <--H2D GO-WritePull----- Host
      Device --D2H Data (64B)-----> Host

**重要限制**:

- 多个 Evict 到同一缓存行不允许
- 多个写请求到同一缓存行允许，但建议设备每次只发出一个 outstanding Write 请求
- 主机在收到先前 snoop 的所有响应和数据前不得发送第二个 snoop
- 设备必须维护请求顺序，因为主机可以自由重排序

CXL.mem 关键流程
-----------------

1. **M2S 请求流程**:
   - Req: 无数据请求 (如内存读取)
   - RwD: 带数据请求 (如内存写入)
   - BIRsp: 对 BISnp 的响应

2. **S2M 响应流程**:
   - NDR: 无数据响应 (如完成确认)
   - DRS: 带数据响应 (如读取数据)
   - BISnp: 反向无效 Snoop (CXL 3.0 新增)

3. **BISnp 支持**:
   - Type 2 设备必须支持 BISnp 通道
   - 允许 Type 2 设备实现包含式 Snoop Filter
   - 支持直接 peer-to-peer 访问 HDM

Fabric 关键流程
----------------

1. **地址解码流程**:
   ::
   
      HPA[51:36] -> 比较 FBase/FLimit -> 确定是否为 Fabric 地址
      -> GlblAddrLkupIdx = HPA[51:36] >> n
      -> 查表获取 DPID 和 IntLvMask
      -> 如果交错启用，二次查找获取 EffDPID

2. **GFAM 访问保护流程**:
   ::
   
      请求地址 -> GTR/GTC 查找 -> 获取 GrpID
      请求 SPID -> SAT 查找 -> 获取 GrpAccVec
      GrpID 选择 GrpAccVec 位 -> 确定是否允许访问

3. **PBR 消息转换**:
   - HBR 到 PBR: 添加 12 位 SPID/DPID，删除 4 位 CacheID/LD-ID
   - PBR 到 HBR: 映射 SPID 到 CacheID，删除 SPID/DPID，添加 LD-ID

4. **Fabric 绑定/解绑**:
   - FM 发布 Bind vPPB 命令指定物理端口、VCS ID 和 vPPB 索引
   - 交换机通知 FM 绑定完成通过生成 Virtual CXL Switch Event Record
   - FM 发布 Unbind vPPB 命令指定 VCS ID 和 vPPB 索引
   - 交换机发起 Managed Hot-Remove 或 Surprise Hot-Remove
